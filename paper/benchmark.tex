\title{AIBugBench: Are AI Bugs Different from Human Bugs? A Human-Paired Benchmark of LLM-Generated Code Defects}

\maketitle

\begin{abstract}
LLM(Large Language Model)이 작성한 코드가 교육과 산업 전반으로 빠르게 들어오면서, 코드 결함을 만드는 주체가 사람에서 LLM으로 이동하고 있다. 이에 따라 LLM이 만드는 버그가 사람 버그와 같은 분포인지, 사람 버그를 전제로 설계된 기존 fault localization과 program repair 도구가 LLM 버그에도 효과적인지가 중요한 질문이 되었다. 그러나 기존 코드 벤치마크는 대체로 손으로 만든 합성 과제나 임의로 선택된 데이터에 의존하며, 코드의 pass/fail 정확도만 측정할 뿐 어떤 종류의 결함이 발생하고 그것이 자동으로 탐지·수리될 수 있는지는 특징화하지 않는다. 더 근본적으로, LLM 버그가 사람 버그와 다른지를 타당하게 비교하려면 생성 모델 입장에서 data leakage가 없으면서 동시에 비교 대상 사람 코드가 진정해야 하지만, 전자는 최신 문제를 후자는 과거 코드를 요구하여 충돌한다. 그 결과 이러한 비교를 가능하게 하는 결함 벤치마크가 부재하다. 본 논문은 data leakage가 없고 같은 문제에서 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench를 제시한다. 핵심 설계는 생성 모델의 knowledge cutoff와 대화형 LLM 코딩 보조 도구의 대중적 등장 사이의 겹치는 구간을 선택하여, 대상 문제는 모델이 학습하지 못했고 그 시기의 사람 제출은 진정함을 동시에 보장하는 것이다(구체적 모델과 시기는 방법론에서 기술한다). AIBugBench는 428개 문제 각각에 대해 정규화된 자연어 명세, 전체 hidden test, LLM이 생성한 10개의 솔루션과 채점 라벨, 그리고 같은 문제에서 짝지어진 사람 정답과 사람 버그(fault location 포함)를 제공한다. open coding으로 결함을 4개 defect type과 11개 sub type으로 분류하고 두 평가자의 독립 분류로 신뢰도를 보고한다. 특징 분석에서 결함의 76\%가 Spec-Misinterpretation이며 Hallucinated-API 같은 고전적 범주는 거의 없었고, 한 문제에 대한 버그가 같은 유형으로 강하게 수렴함을 확인하였다. 또한 기존 교육용 program repair 도구가 이 결함 분포로 전이되지 못함을 보인다. 데이터셋, 코드, 재현 파이프라인을 공개한다.
\end{abstract}

\section{Introduction}
LLM(Large Language Model)이 작성한 코드가 교육과 산업 전반으로 빠르게 확산되고 있다. 프로그래밍 교육에서는 학생이 과제 코드를 LLM으로 생성해 제출하는 일이 흔해졌고, 산업에서는 코딩 보조 도구가 일상화되었다~\cite{chen2021codex,tambon2025bugs}. 그 결과 코드 결함을 만드는 주체가 사람에서 LLM으로 옮겨가고 있으며, 이는 두 가지 실천적 질문을 제기한다. 첫째, LLM이 만드는 버그는 사람 버그와 같은 분포인가? 둘째, 사람 버그를 전제로 설계된 기존 fault localization과 APR(Automatic Program Repair) 도구가 LLM 버그에도 효과적인가? 이 질문들에 답하려면 LLM 코드 결함을 특징화하고 같은 문제에서 사람 버그와 비교할 수 있는 데이터가 필요하다.

이 질문의 답은 단지 학술적 호기심을 넘어선다. 교육 현장에서는 교수자가 학생이 제출한 LLM 코드의 결함을 진단하고 피드백을 주어야 하며, 결함의 유형과 그 수리 가능성을 알아야 적절한 도구와 교습 전략을 선택할 수 있다. 산업에서는 코딩 보조 도구가 만든 코드가 코드베이스에 직접 들어오므로, 그 결함 분포를 이해하는 것이 코드 리뷰와 테스트 자동화의 우선순위를 정하는 데 도움이 된다. 그러나 잘못된(예: contamination된) 데이터로 도출한 결론은 이러한 의사결정을 오도할 수 있다.

문제를 더 어렵게 만드는 것은 LLM 결함의 은밀함이다. LLM이 만든 의미적 버그는 컴파일되고 실행되며 공개 예제 테스트를 통과하기도 하므로, 사람 리뷰어나 자동 도구가 ``정상''으로 오판하기 쉽다. 예컨대 명세의 한 조건을 빠뜨린 코드는 예제는 통과하지만 숨은 입력에서만 실패한다. 이런 결함이 교육 과제나 코드베이스에 누적되면, 사람 버그를 전제로 조정된 fault localization과 repair 파이프라인이 이를 놓치거나 잘못 수리할 수 있다. 따라서 LLM 결함의 분포와 수리 가능성을 정량화하는 일은 단순한 측정이 아니라, LLM 시대의 테스트·디버깅·교육 도구를 어디에 투자할지 결정하는 실질적 토대가 된다.

그러나 기존 코드 벤치마크는 이 목적에 잘 맞지 않는다. HumanEval~\cite{chen2021codex}과 MBPP~\cite{austin2021mbpp}는 손으로 만든 합성 과제로 사람 버그가 없고, 코드가 맞는지/틀리는지의 pass/fail만 측정할 뿐 어떤 종류의 결함이 왜 발생하는지는 특징화하지 않는다. EvalPlus~\cite{liu2023evalplus}는 테스트를 보강하면 LLM 코드의 통과율이 크게 하락함을 보였으나 여전히 실패를 분류하지 않으며, 공개 벤치마크의 contamination 문제도 보고된다~\cite{golchin2024contamination}. Defects4J~\cite{just2014defects4j}, CodeNet~\cite{puri2021codenet}, ConDefects~\cite{wu2023condefects} 같은 결함 데이터셋은 사람 버그를 담지만 AI 생성 버그를 담지 않는다. 따라서 LLM 코드의 결함을 분류하고 기존 도구로 평가할 표준 데이터가 없다.

더 근본적으로, ``LLM 버그가 사람 버그와 다른가''를 타당하게 비교하려면 두 가지 교란을 제거해야 한다. 첫째는 data leakage다. 대상 문제가 모델 학습 데이터에 이미 있으면 생성은 추론이 아니라 암기 재현이 되어, 관찰된 결함이 모델의 실제 추론 특성을 대표하지 못한다. 둘째는 사람 대조군 오염이다. LLM이 대중화된 이후의 사람 제출은 그 자체가 AI 보조의 산물일 수 있어 ``사람'' 대조군의 진정성이 훼손된다. 누수를 피하려면 최신(모델 cutoff 이후) 문제를, 진정한 사람 코드를 얻으려면 과거(LLM 대중화 이전) 코드를 써야 하므로, 같은 문제 집합에서 두 요구가 충돌한다. \textbf{그 결과, data leakage가 없으면서 진정한 사람 버그와 짝지어진 LLM 결함 벤치마크는 부재하다.}

이 공백을 메우는 벤치마크는 세 조건을 동시에 만족해야 한다. 첫째 AI 생성 버그를 포함하고, 둘째 생성 모델 입장에서 leakage가 없으며, 셋째 같은 문제에서 진정한 사람 버그와 짝지어져야 한다. \textbf{현존하는 벤치마크 중 이 세 조건을 모두 만족하는 것은 없다.}

본 논문은 leakage가 없고 사람과 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench를 제시한다. 핵심 관찰은 두 제약이 서로 다른 산출물에 서로 다른 시점으로 걸린다는 것이다. leakage는 생성 대상 문제에(모델 cutoff 기준), 사람 진정성은 사람 제출에(LLM 대중화 시점 기준) 걸린다. 두 시점이 다르므로 그 사이에 두 조건이 동시에 성립하는 겹치는 구간이 존재하며, 본 논문은 이 구간을 선택해 leakage-free와 human-authentic을 동시에 달성한다. 구체적 생성 모델과 시기 경계는 방법론에서 기술한다.

AIBugBench를 구축하기 위해, ConDefects~\cite{wu2023condefects}에서 이 구간의 Python 과제 428개를 선별하고, 각 과제의 자연어 명세를 AtCoder에서 수집해 정규화하였으며, 전체 hidden test를 확보하였다. 각 문제에 대해 LLM으로 10개의 솔루션을 생성하여 전체 테스트로 채점하고, 실패한 코드를 사람 정답과 대조하는 open coding으로 분류 체계를 도출하였다. Defects4J~\cite{just2014defects4j}와 ConDefects의 정신을 따라, 버그와 정답 버전, 테스트, 명세를 손쉽게 가져와 실행할 수 있는 재현 파이프라인을 함께 제공한다.

본 연구에서 답하는 연구 질문은 다음과 같다.

\noindent\textbf{RQ1: LLM 생성 프로그램에는 어떤 종류의 결함이 흔히 나타나는가?} 3{,}691개 버그를 분석한 결과 결함의 76\%가 Spec-Misinterpretation으로, 문법적으로 정상이고 실행되지만 다른 문제를 푸는 코드였다(하위 유형 중 핵심 조건 단순화 S2 57\%와 패러다임 미인식 S4 33\%가 90\%를 차지). 비효율로 인한 TLE(Time Limit Exceeded)가 14\%, RE(Runtime Error)가 9\%, CE(Compile Error)가 1\%였고 Hallucinated-API는 사실상 없었다.

\noindent\textbf{RQ2: LLM 버그는 사람 버그와 어떻게 다른가?} 같은 문제에서 짝지어진 사람 버그와 분포가 유의하게 다르며($\chi^2=139.8$, $p<10^{-29}$), 사람 버그는 100\%가 단일 라인의 국소적 슬립인 반면 AI 버그는 프로그램 전체가 잘못된 접근이었다.

\noindent\textbf{RQ3: 한 문제에 대한 LLM의 버그는 무작위인가 systematic한가?} 버그가 2개 이상인 387개 문제에서 평균 modal-share가 0.845, 완전 일치 문제 비율이 38.8\%로, LLM은 한 문제를 일관되게 같은 방식으로 오해하였다.

\noindent\textbf{RQ4: 난이도에 따라 벤치마크는 얼마나 도전적이며 결함 프로파일은 어떻게 변하는가?} 전체 통과율은 13.8\%이고 어려운 문제에서는 0\%였으며, 어려워질수록 패러다임 미인식(S4)의 비중이 WA의 8\%에서 약 40\%로 증가하였다.

\noindent\textbf{RQ5: 기존 수리 도구는 이러한 LLM 버그를 얼마나 다룰 수 있는가?} 교육용 cluster 기반 수리는 본 분포로 전이되지 못하였고, LLM 기반 수리도 실패 테스트를 주어도 교차 모델 10.7\%·자기 수리 1.1\%에 그쳤다. 우리는 기존 도구가 실패하는 다섯 가지 근본 한계를 분석한다.

본 논문의 기여는 다음과 같다.
\begin{itemize}
\item leakage가 없고 같은 문제에서 진정한 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench.
\item 428개 문제, 4{,}280개 AI 생성물(문제당 10개)과 채점 라벨, 1{,}204개의 짝지어진 사람 버그(fault location 포함), 정규화 명세, 전체 hidden test로 이루어진 데이터셋과, 4개 defect type 및 11개 sub type의 분류 체계.
\item 벤치마크가 지원하는 다섯 평가 태스크와 측정 지표, 그리고 표준 분할과 평가 스크립트로 이루어진 평가 프로토콜.
\item 생성 모델의 난이도별 통과율과 결함 분포, systematicity를 reference 특성으로 보고하고, 교육용 수리 도구의 전이 실패와 그 근본 한계를 분석.
\item 데이터, 코드, 평가 스크립트, 재현 파이프라인을 담은 replication package를 공개한다(이중 익명 심사를 위해 링크는 익명화하며, 검토용 익명 아카이브를 보충 자료로 제출한다).
\end{itemize}

본 논문의 새로움은 데이터의 규모가 아니라 그 구성 원리에 있다. 최근 연구들이 HumanEval 같은 공개 벤치마크에서 LLM 코드 오류를 분류하였으나~\cite{wang2025characteristics,tambon2025bugs,dou2026wrong}, 그 벤치마크들은 생성 모델 입장에서 오염 가능성이 있고 같은 문제의 사람 대조군이 없다. 본 논문은 처음으로 (i) 생성 모델 cutoff 이후이자 LLM 대중화 이전이라는 이중 제약을 만족하는 시기 윈도우를 도입하여 data leakage와 사람 대조군 오염을 동시에 제거하고, (ii) 같은 문제에서 AI와 사람 결함을 짝지어 ``다른가''를 통계적으로 검정하며, (iii) 명세·전체 테스트·라벨·사람 정답을 묶어 재사용 가능한 실행 벤치마크로 패키징한다. 이 세 요소의 결합이 기존 연구와의 본질적 차이이며, 어느 하나만으로는 ``AI 버그는 사람 버그와 다른가''라는 질문에 타당하게 답할 수 없다.

\section{Background and Related Work}
\subsection{LLM Code Generation and Its Evaluation}
Codex와 그 평가 벤치마크 HumanEval~\cite{chen2021codex}, 그리고 MBPP~\cite{austin2021mbpp} 이후 LLM 기반 코드 생성은 빠르게 발전하였고, 표준 지표로 pass@$k$가 자리 잡았다. 그러나 이 지표는 코드가 맞는지/틀리는지만 보고하고 어떤 종류의 결함이 왜 발생하는지는 알려주지 않는다. 더욱이 통과/실패 판정은 테스트 스위트의 강도에 민감하다. EvalPlus~\cite{liu2023evalplus}는 HumanEval의 테스트를 대폭 보강하면 여러 모델의 통과율이 크게 하락함을 보여, 빈약한 테스트가 정확도를 과대평가함을 입증하였다. 최근에는 SWE-bench~\cite{jimenez2024swebench}처럼 실세계 GitHub 이슈 해결 능력을 평가하는 벤치마크도 등장했으나, 이 역시 통과/실패 중심이고 공개 저장소를 기반으로 하여 contamination 우려가 있으며 결함을 분류하지 않는다. 이러한 한계 때문에, 단순 정확도를 넘어 ``어떤 종류의 결함이 왜 발생하는가''를 묻는 결함 중심 평가가 필요하다. AIBugBench는 이 관찰을 결함 분석 맥락에서 재현하면서(공개 sample이 아닌 전체 hidden test 사용), 통과/실패를 1차 신호로만 쓰고 실패한 코드를 의미 단위로 분류하여 평가의 해상도를 높이며, 같은 문제에서 사람 버그와 짝짓는다는 점에서 통과율 중심 평가 벤치마크와 근본적으로 구분된다.

\subsection{Empirical Studies of Bugs in LLM-Generated Code}
LLM 코드의 결함을 경험적으로 분석한 연구가 본 논문과 가장 가깝다. 특히 Wang 등~\cite{wang2025characteristics}은 HumanEval에서 여섯 LLM의 코드 생성 오류를 open coding으로 분석하여 의미·구문 양면의 결함 분류 체계를 도출하고, 오류가 여러 위치에 걸친 비자명한 다중 라인 결함이라 탐지·수리가 어렵다고 보고하였다―이는 본 논문의 ``AI 버그는 한 줄로 환원되지 않는다''는 관찰과 부합한다. Tambon 등~\cite{tambon2025bugs}과 Dou 등~\cite{dou2026wrong}은 여러 모델·벤치마크에 걸쳐 LLM 코드 오류를 대규모로 분류하였고, Jesse 등~\cite{jesse2023bugs}은 LLM이 사람과 유사한 단순 버그(simple, stupid bugs)를, Pan 등~\cite{pan2024lost}은 코드 번역 과정에서 LLM이 도입하는 버그를 연구하였다. Pearce 등~\cite{pearce2022asleep}과 Moradi Dakhel 등~\cite{dakhel2023github}은 Copilot 코드의 보안 취약점과 ``자산이냐 부채냐''라는 양면을 분석하였고, 교육 맥락에서는 Yeadon 등~\cite{yeadon2024comparison}이 사람과 GPT-3.5·GPT-4의 코딩 성능을 비교하여 LLM이 사람 수준에 근접·초과함을 보였다.

이들 연구는 LLM 결함의 존재·다양성·교육적 함의를 풍부하게 보였으나, 본 논문과 세 가지 점에서 구별된다. 첫째, 평가 기반이 대체로 HumanEval 등 공개·합성 벤치마크여서 생성 모델의 data leakage가 통제되지 않으며(예: Wang 등은 HumanEval을 사용한다), 관찰된 결함이 추론과 암기 중 무엇을 반영하는지 모호하다. 둘째, 같은 문제에서 진정한 사람 버그와 짝지어 비교하지 않으므로 ``AI 버그가 사람 버그와 다른가''를 직접 검정하지 못한다. 셋째, 분석이 일회적 연구에 그쳐 후속 도구를 개발·평가할 재사용 가능한 실행 벤치마크로 공개되지 않으며, 한 문제 안에서 여러 생성 버그가 같은 오해로 수렴하는지(systematicity)도 다루지 않는다. AIBugBench는 leakage-free·human-authentic 구간에서 같은 문제의 사람 버그와 짝지어, 결함 분류·systematicity·사람 대비 분포 차이·수리 baseline을 표준 분할·평가 스크립트와 함께 제공함으로써 이들 연구를 재사용 가능한 벤치마크로 보완·확장한다.

\subsection{Data Leakage and Contamination}
공개 벤치마크가 학습 데이터에 포함되는 contamination은 LLM 평가의 타당성을 훼손하며, 이를 탐지하고 정량화하려는 연구가 활발하다~\cite{golchin2024contamination}. 코드 생성에서 contamination은 특히 심각한데, 모델이 정답을 암기했다면 측정된 능력이나 결함 특성이 실제 추론을 대표하지 못하기 때문이다. 모델 cutoff 이후에 출제된 문제만 사용하는 것이 가장 직접적인 완화책이며, ConDefects~\cite{wu2023condefects}는 이 원리로 fault localization과 program repair 평가의 leakage를 통제하였다. 특히 코드 도메인에서는 모델이 특정 문제의 정답이나 그 패치까지 기억할 수 있어, 결함의 종류를 분석하는 일은 물론 수리 능력을 측정하는 일까지 왜곡될 수 있다. 따라서 결함·수리 벤치마크에서 leakage 통제는 단순한 위생 문제가 아니라 결과의 해석 가능성을 좌우하는 핵심 설계 결정이다. 본 논문은 여기서 한 걸음 더 나아가, 생성 모델의 leakage뿐 아니라 사람 대조군의 진정성까지 동시에 보장하는 시기 윈도우를 도입함으로써, ``AI 버그가 사람 버그와 다른가''라는 비교가 두 교란 어느 쪽에도 오염되지 않도록 한다.

\subsection{Automatic Program Repair and Fault Localization}
APR은 탐색 기반, 의미 기반, 학습 기반으로 발전해 왔으며~\cite{monperrus2018survey}, 최근에는 LLM을 직접 수리에 활용하는 접근이 강력한 성능을 보인다~\cite{xia2023llmapr}. 수리의 전 단계인 fault localization은 의심 위치를 순위화하는 오랜 연구 주제로, 다양한 기법과 평가 지표가 정립되어 있다~\cite{wong2016flsurvey}. 이들 도구는 대체로 버그가 사람에 의해 작성되었다고 가정하며, 그 결함 분포―흔히 한두 줄의 국소적 결함―에 맞춰 설계되거나 학습된다. 그러나 LLM 버그가 국소 슬립이 아니라 프로그램 전체의 의도 오류라면, 단일 위치를 가정하는 fault localization과 국소 변형을 가정하는 수리 연산자가 모두 불리해진다. AIBugBench는 이들 도구를 LLM 버그에 대해 평가할 수 있는 기반을 제공하고, ConDefects가 제공하는 사람 fault location을 fault localization의 정답으로 함께 제공하여 사람 버그와 AI 버그에서의 도구 정확도를 같은 자리에서 대조할 수 있게 한다.

\subsection{Program Repair for Programming Education}
교육용 입문 프로그래밍 수리에서는 같은 과제에 대한 다수의 정답 풀이를 활용하는 접근이 주류다. Clara~\cite{gulwani2018clara}는 정답을 클러스터링하여 학생 코드를 가장 가까운 정답으로 수리하고, Refactory~\cite{hu2019refactory}는 정답 풀이 군을 control flow에 따라 재구성하여 버기 프로그램에 맞춰 패치를 합성하며, Verifix~\cite{ahmed2022verifix}는 참조 풀이에 대해 검증된 수리를 생성한다. 이들은 사람 버그와 풍부한 정답 풀이 풀을 전제로 한다는 공통점이 있다. 본 연구의 결과는 LLM 버그가 의미적으로 틀리고 풀이가 다양한 경쟁 난이도 분포를 가져, 이들 도구의 가정과 어긋날 수 있음을 시사한다.

\subsection{Defect Benchmarks}
Defects4J~\cite{just2014defects4j}는 실세계 Java 결함의 버기/패치 버전을 손쉽게 체크아웃해 controlled study를 가능하게 한 대표적 실행 벤치마크로, 이후 수많은 도메인에서 유사한 벤치마크의 모범이 되었다. QuixBugs~\cite{lin2017quixbugs} 같은 소규모 수리 벤치마크는 단일 라인 버그를 담지만 규모가 작고 AI 생성 버그가 없다. CodeNet~\cite{puri2021codenet}과 Codeforces 계열은 사람 제출(정답과 오답)을 대규모로 제공하나 AI 생성 버그가 없고 LLM 이전에 구축되어 생성 모델 입장에서 leakage가 있다. ConDefects~\cite{wu2023condefects}는 모델 cutoff 이후의 AtCoder 문제만 모아 leakage를 통제하였으나 자연어 명세와 AI 생성 버그를 포함하지 않는다. \textbf{즉 LLM 생성 버그를, leakage가 없는 동시에 진정한 사람 버그와 짝지어 실행 가능한 형태로 제공하는 결함 벤치마크는 부재하며, AIBugBench가 이 공백을 메운다.}

\subsection{Selecting the Leakage-Free, Human-Authentic Window}
생성 모델로는 knowledge cutoff가 잘 문서화되어 있고 그 cutoff가 대화형 코딩 보조 도구의 대중적 등장보다 앞서는 모델이 필요하다. 우리는 \texttt{gpt-3.5-turbo}(knowledge cutoff 2021년 9월)를 고정한다. data leakage가 없으려면 대상 문제가 2021년 9월 이후 출제여야 한다. 한편 사람 대조군이 진정하려면 사람 제출이 LLM이 대중화되기 전에 작성되어야 하는데, 우리는 대중화의 분기점으로 ChatGPT의 공개 출시(2022년 11월 30일)를 택한다. ChatGPT는 학생과 개발자가 코드 생성을 위해 LLM을 일상적으로 쓰기 시작한 가장 뚜렷한 변곡점이기 때문이다. 두 시점이 다르므로 그 사이의 [2021-09, 2022-11] 구간에서 두 조건이 동시에 성립한다. 이 구간에 출제된 문제는 모델이 학습하지 못했고, 그 컨테스트의 사람 제출은 ChatGPT 이전이므로 진정하다. 다만 사람 진정성은 ``컨테스트 시기''라는 proxy에 의존하며, 옛 문제에 대한 후대 재제출 가능성은 한계로 남는다(Section~VIII).

이 설계의 타당성은 두 cutoff의 분리에 달려 있다. 형식적으로, 문제 출제 시점을 $t_p$, 생성 모델의 knowledge cutoff를 $t_c$, 사람 제출 시점을 $t_h$, LLM 대중화 시점을 $t_r$라 하자. leakage-free는 $t_p > t_c$를, human-authentic은 $t_h < t_r$를 요구한다. 한 컨테스트에서 출제와 사람 제출은 거의 동시에 일어나 $t_p \approx t_h$이므로, 두 조건은 $t_c < t_p \approx t_h < t_r$로 묶이고, 따라서 $t_c$와 $t_r$ 사이가 둘을 동시에 만족하는 유일한 창이다. 이 창이 비어 있지 않으려면 $t_c < t_r$, 즉 모델의 cutoff가 LLM 대중화보다 앞서야 하며, 우리가 고른 모델 계열이 정확히 이 성질을 갖는다. 이 논리는 특정 모델에 묶이지 않으며 cutoff가 $t_r$ 이전인 모든 모델에 동일하게 적용되므로, 벤치마크는 향후 그러한 모델로 자연스럽게 확장된다.

\subsection{Problem Selection from ConDefects}
ConDefects는 각 과제를 \texttt{Code/<task>/<lang>/<id>/} 아래에 사람 버기 버전, 그에 대응하는 사람 정답, 그리고 fault location으로 제공한다. 우리는 다음 기준으로 문제를 선별한다: (i) Python 과제일 것, (ii) contest date가 시기 윈도우 안일 것, (iii) 사람 정답과 버기 짝이 모두 존재할 것, (iv) 전체 hidden test가 확보될 것. AtCoder는 2021년 후반부터 Beginner Contest의 마지막 문제 표시 라벨을 ``Ex''로 바꾸었는데, ConDefects는 이를 \texttt{\_h}로 표기한다. 이 불일치 때문에 테스트 디렉터리를 \texttt{\_h}로만 찾으면 22개 문제의 전체 테스트를 놓치므로, \texttt{\_h}를 ``Ex''로 매핑하여 복구하였다. 또한 표준 stdin/stdout 채점이 불가능한 interactive 문제 4개(예: abc244\_c, abc269\_e)는 제외하였고, 복수 정답을 허용하는 special judge 문제는 제거하지 않되 \texttt{meta.json}에 표시하여 사용자가 선택적으로 처리하게 한다. 필터링은 단계적으로 적용된다: 윈도우 내 AtCoder 과제에서 출발하여 Python 풀이가 없는 과제를 제거하고, 사람 정답·버기 짝이 모두 존재하지 않는 과제를 제거하며, 전체 hidden test를 확보하지 못한 과제를 제거한다. 이 과정을 거쳐 ABC 330·ARC 92·AGC 6개로 구성된 85개 컨테스트의 428개 문제와, 그에 짝지어진 1{,}204개의 사람 버기 프로그램(및 동수의 사람 정답과 fault location)을 얻는다.

\subsection{Specifications}
ConDefects는 코드와 테스트는 제공하나 자연어 명세는 포함하지 않으므로, 각 과제의 영어 명세를 AtCoder 문제 페이지에서 수집한다. 명세는 \texttt{task-statement} 영역의 영어 블록에 있으며, 문제 설명·제약·입출력 형식·예시로 구성된다. 단순히 HTML에서 텍스트를 추출하면 수식 변수가 토큰마다 줄바꿈되어 가독성이 크게 떨어지므로, 우리는 문단과 예시 블록을 구조적으로 재구성하는 정규화를 적용한다: 문단은 한 줄로 합치고 예시 입출력 블록은 원형을 보존한다. AtCoder 문제 페이지는 영어와 일본어 명세를 함께 담으므로 \texttt{lang-en} 블록만 선택하고, MathJax 수식은 평문으로 평탄화하되 변수 기호는 보존한다. 예제 입출력은 채점 형식과 정확히 일치해야 하므로 원형 그대로 유지한다. 정규화 전후를 비교하면, 단순 텍스트 추출은 한 수식을 여러 줄로 쪼개 명세를 사실상 읽을 수 없게 만드는 반면 본 정규화는 사람이 읽을 수 있는 문단을 복원한다. 정규화된 명세는 \texttt{problem.txt}로 제공되며 생성과 spec-grounded repair의 입력이 된다.

\subsection{Tests and Execution Environment}
채점에는 ConDefects의 전체 hidden test를 사용한다. 솔루션은 각 테스트에 대해 stdin을 주입하고 stdout을 기대 출력과 비교하며, 출력 비교는 줄 단위 후행 공백과 마지막 개행을 무시하는 관용 비교를 따른다. 실행은 격리된 subprocess에서 이루어지고, 문제별 시간 제한과 메모리 상한을 적용하며 네트워크를 차단한다. AtCoder의 표준 라이브러리 가정을 반영하기 위해 채점 샌드박스에 \texttt{sympy}, \texttt{sortedcontainers}, \texttt{numpy}를 설치하여, 라이브러리 부재로 인한 거짓 RE를 방지한다. 공개 sample만으로 채점하면 통과율이 13.8\%가 아니라 23\%로 과대평가되므로, 전체 hidden test가 타당한 채점에 필수임을 확인하였다. 품질 관리를 위해 전체 테스트는 문제별로 \texttt{testcases/}에 materialize하되 입력과 기대 출력이 모두 존재하는 케이스만 채택하였고, 각 생성물의 채점 결과(status와 통과 테스트 수)를 함께 저장하여 재현과 검증이 가능하게 하였다.

\subsection{Generating AI Solutions}
각 문제에 대해 \texttt{gpt-3.5-turbo}로 temperature 1.0, 문제당 10개의 솔루션을 생성한다. 프롬프트는 ``Output ONLY a complete Python 3 program''이라는 간결한 system 지시와 정규화된 명세(user 메시지)로 구성한다. 이 system 지시가 없으면 생성물의 46\%가 C++ 등 비-Python으로 이탈하는데, 이는 AtCoder 문제가 관습적으로 C++ 중심이기 때문이다. system 지시 한 줄로 이를 통제하여 본 설정의 모든 생성물이 100\% Python이도록 한다. 응답에서 코드는 fenced block로 추출하고, 재현성을 위해 모델 응답이 보고하는 정확한 snapshot을 레코드에 함께 저장한다. 문제당 10개를 생성하는 것은 한 문제 안에서 버그가 같은 유형으로 수렴하는지(systematicity)를 측정하기에 충분한 표본을 확보하기 위함이며, temperature 1.0은 결정론적 반복이 아니라 모델의 자연스러운 출력 분포를 관찰하기 위한 선택이다. \texttt{gpt-3.5-turbo}는 저렴하고 빠르므로 문제당 10개 생성의 비용은 작아, 전체 4{,}280개 생성을 적은 비용으로 확보하였다.

\subsection{Judging and Deriving the Taxonomy}
각 솔루션을 전체 hidden test로 채점하여 pass와 WA(Wrong Answer), TLE(Time Limit Exceeded), RE(Runtime Error), CE(Compile Error)로 분류한다. 결함 분류 체계는 사전 범주 없이 open coding으로 도출한다. TLE, RE, CE는 채점 status와 예외 정보로 직접 태깅한다. 의미 판단이 필요한 WA는 실행은 되나 출력이 틀린 경우로, 두 저자가 각 AI 버그를 같은 문제의 사람 정답(\texttt{correct.py})과 대조하여 무엇을 오해했는지 파악하고, 유사한 오해를 반복적으로 병합하여 하위 유형을 만든다. 이 과정에서 Spec-Misinterpretation은 다섯 하위 유형(S1--S5)으로, RE는 세 하위 유형으로 세분되어 전체 4개 defect type과 11개 sub type의 분류 체계가 도출된다(Section~VI-A). 범주가 확정된 뒤에는 전체 2{,}808개 WA 버그를 생성 모델과 다른 모델(\texttt{gpt-4o-mini})을 주석자로 사용하여 일관되게 하위 유형으로 라벨링하였고, 신뢰도는 독립적인 두 번째 주석 패스와의 Cohen's $\kappa$~\cite{cohen1960kappa}로 측정하였다. 300개 표본에서 $\kappa=0.759$(원시 일치도 0.86)로 목표 $\kappa\geq 0.7$을 만족하였다.

\subsection{Systematicity Metric}
LLM 버그가 무작위인지 systematic한지 평가하기 위해, 한 문제에 대한 여러 생성의 버그가 같은 실패 유형으로 수렴하는 정도를 정량화한다. 문제 $p$의 버그들을 범주 $c$별로 센 분포를 $\{n_c\}$, 버그 수를 $N_p=\sum_c n_c$, 서로 다른 범주 수를 $k_p$라 하자. modal-share를
\[
m_p \;=\; \frac{\max_c n_c}{N_p}
\]
로 정의한다. $m_p=1$이면 모든 버그가 같은 유형(완전 systematic)이다. 또한 분포의 정규화 Shannon entropy를
\[
H_p \;=\; \frac{-\sum_c (n_c/N_p)\,\log_2 (n_c/N_p)}{\log_2 k_p}
\]
로 정의하며($k_p>1$일 때), $H_p=0$이면 완전 systematic, $H_p=1$이면 균등 분산이다. 버그가 2개 이상인 문제에 대해 $m_p$와 $H_p$를 집계하여 systematicity를 보고한다.

\subsection{Benchmark Format and Usage}
AIBugBench는 문제마다 자족적인 디렉터리로 배포되어, Defects4J처럼 단일 항목을 손쉽게 가져와 실행할 수 있다. 각 문제 디렉터리는 정규화 명세(\texttt{problem.txt}), 메타데이터(\texttt{meta.json}; 난이도·컨테스트·special-judge 표시 포함), 전체 hidden test(\texttt{testcases/}), 문제당 10개의 AI 생성물과 채점 라벨(\texttt{ai/gen\_$k$.py}, \texttt{ai/results.json}), 그리고 짝지어진 사람 항목(\texttt{human/<id>/}의 \texttt{faulty.py}, \texttt{correct.py}, \texttt{faultLocation.txt})으로 구성된다.

\begin{lstlisting}
data/<task_id>/
  problem.txt              # normalized NL specification
  meta.json                # difficulty, contest, special-judge flag
  testcases/               # full hidden tests (stdin/stdout)
  ai/ gen_1.py ... gen_10.py
      results.json         # per-generation status + n_passed/n_tests
  human/<submission_id>/
      faulty.py  correct.py  faultLocation.txt
\end{lstlisting}

이 구조에서 AI 버그 분류는 \texttt{ai/}와 \texttt{problem.txt}를, fault localization은 \texttt{human/}의 \texttt{faultLocation.txt}를, program repair는 \texttt{testcases/}와 \texttt{correct.py}를, AI 대 사람 비교는 \texttt{ai/}와 \texttt{human/}을 직접 사용한다. 제공된 채점 하네스는 동일한 시간 제한·라이브러리 환경에서 \texttt{testcases/} 전체로 패치를 재채점하여 결과를 비교 가능하게 한다.

\section{Supported Tasks and Evaluation Protocol}
AIBugBench는 다섯 태스크를 지원하며, 각 태스크의 입력·정답·측정 지표를 표준화하고 본 논문의 결과를 reference로 제공한다.
\begin{itemize}
\item \textbf{Bug classification}: 입력은 AI 버그(코드, 명세, 채점 결과), 출력은 결함 범주(4 defect type / 11 sub type)이며, accuracy와 주석자 간 Cohen's $\kappa$로 측정한다. 본 논문의 $\kappa=0.759$가 reference다.
\item \textbf{Fault localization}: 입력은 버기 프로그램과 테스트, 출력은 의심 라인 순위이며, 사람 \texttt{faultLocation.txt}를 정답으로 top-$k$ accuracy와 EXAM score로 측정한다~\cite{wong2016flsurvey}. AI 버그가 단일 라인으로 환원되지 않는다는 점은 라인 기반 지표의 한계를 시험하는 무대가 된다.
\item \textbf{Automatic program repair}: 입력은 버기 프로그램·전체 테스트·참조 정답, 출력은 패치이며, 전체 테스트를 통과하는 plausible 패치율과 정답과 동치인 correct 패치율을 구분해 측정한다~\cite{monperrus2018survey,xia2023llmapr}. 본 논문의 self 1.1\%/cross 10.7\%가 reference baseline이다.
\item \textbf{AI versus human comparison}: 입력은 같은 문제의 AI·사람 버그, 출력은 범주 분포 비교이며, $\chi^2$ 검정과 난이도별 교차표로 측정한다. 본 논문의 $\chi^2=139.8$이 reference다.
\item \textbf{Spec-grounded repair}: 입력은 명세와 버기 프로그램이며, 명세에서 추출한 의미적 제약을 테스트와 별개의 두 번째 oracle로 활용하여 Spec-Misinterpretation(특히 S2/S4) 한정 수리율 향상으로 측정한다.
\end{itemize}

재현 가능한 평가를 위해 표준 분할과 실행 하네스를 함께 제공한다. 표준 분할은 문제 단위로 train/validation/test를 약 70/15/15로 나누어(대략 300/64/64 문제), 같은 문제의 솔루션이 분할을 넘나들지 않게 함으로써 누수 없는 평가를 보장한다. 모든 채점은 제공된 평가 스크립트와 동일한 시간 제한 및 라이브러리 환경에서 수행하여 결과를 비교 가능하게 한다.

\section{Evaluation}
이 절은 서론의 다섯 연구 질문에 답한다. RQ1과 RQ2는 결함의 종류와 사람 대비 차이를, RQ3은 그 systematicity를, RQ4는 난이도에 따른 도전성과 결함 프로파일을, RQ5는 기존 도구의 처리 능력을 다룬다.

\subsection{RQ1: What Types of Bugs Do LLMs Produce?}
표~\ref{tab:taxonomy}는 3{,}691개 버그의 분류 분포이다. Spec-Misinterpretation이 2{,}808개(76\%)로 압도적이며, 다섯 하위 유형으로 세분된다: S1 출력/케이스 단순화(예: ``er''/``ist''를 항상 끝 두 글자로 출력), S2 핵심 조건 단순화(예: 칸 connectivity를 칸 존재 여부로), S3 잘못된 알고리즘/공식(예: 중복 순열을 일반 순열 공식으로), S4 패러다임 미인식(예: topological sort를 완전 탐색으로), S5 엣지/언어 의미(예: \texttt{round()}의 banker's rounding). 비효율로 인한 TLE가 504개(14\%), RE가 338개(9\%, 입력 형식 오독·인덱스 경계·수치/환경으로 세분), 출력 잘림 등 CE가 41개(1\%)였다. 특히 Hallucinated-API는 사실상 없었고, 관찰된 \texttt{ModuleNotFoundError}는 환각이 아니라 미설치 라이브러리 때문이었다. 2{,}808개 WA를 전수 라벨링하면($\kappa=0.759$) S2 핵심 조건 단순화가 57.3\%(1{,}609개), S4 패러다임 미인식이 32.7\%(918개)로 둘이 90\%를 차지하고, S3(6.5\%), S1(1.9\%), S5(1.6\%)는 드물었다.

\rabox{\textbf{RA1.} LLM 생성 결함은 76\%가 Spec-Misinterpretation으로, 문법적으로 정상이고 실행되지만 다른 문제를 푸는 코드다. 비효율(14\%), 런타임 오류(9\%), 출력 잘림(1\%)이 뒤따르며, API 환각은 사실상 없다.}

그림~\ref{fig:example}는 대표적 Spec-Misinterpretation(S2) 버그이다. 문제는 $N$번째 AGC를 0으로 채운 3자리 수로 표기하되 42번 이후는 한 회차가 취소되어 실제 번호가 $N{+}1$이 되는 규칙을 담는다. AI 코드는 문법적으로 정상이고 실행되지만 이 ``42번 이후 +1'' 규칙을 통째로 누락하여 더 쉬운 문제를 푼다. 결함은 한 줄이 아니라 빠진 로직 전체이므로 라인 단위 fault localization으로 짚기 어렵다.

else:         print("AGC" + str(N))
\end{lstlisting}
\begin{lstlisting}

z = 2 if N < 10 else 1
print("AGC" + "0" * z + str(N))
\end{lstlisting}

Spec-Misinterpretation의 두 지배 유형은 서로 다른 실패 방식을 보인다. 그림~\ref{fig:example}의 S2가 한 조건을 빠뜨리는 ``부분 누락''이라면, 그림~\ref{fig:example3}의 S4는 문제가 요구하는 알고리즘 자체를 인식하지 못하는 ``패러다임 미인식''이다. 후자에서 모델은 ``$(K{-}1)$번째 순열을 구하라''를 모든 순열을 나열해 $P$를 제거하는 완전 탐색으로 치환하여, 의도한 $O(N)$ previous-permutation 대신 $O(N!)$ 접근을 택한다. 한편 WA가 아닌 결함도 진단적 의미를 갖는다. 338개 RE의 예외를 재수집하면(표~\ref{tab:re}) 인덱스 경계(153개, 주로 \texttt{IndexError})가 가장 많고, 수치·환경(95개; \texttt{TypeError}·\texttt{RecursionError}·미설치 라이브러리 등), 입력 형식 오독(62개; \texttt{ValueError}·\texttt{EOFError}로 한 줄 다중 정수를 잘못 분해), 기타(28개)가 뒤따른다. CE 41개는 주로 출력 잘림과 형식 오류였다. 즉 본 분류는 ``틀린 답''을 넘어 어떻게 틀렸는지를 구조화한다.

\subsection{RQ2: How Do AI Bugs Differ from Human Bugs?}
본 벤치마크가 가능하게 하는 핵심 비교로, 같은 문제에서 짝지어진 사람 버그 1{,}203개를 동일한 전체 테스트로 채점하여 AI와 분포를 대조하였다(표~\ref{tab:aihuman}). 두 분포는 유의하게 다르다($\chi^2=139.8$, $\mathit{dof}=3$, $p<10^{-29}$). 사람 버그는 91.5\%가 WA로 한 가지 실패에 집중되는 반면 AI는 TLE·RE·CE에서 사람보다 2{--}4배 높아 더 다양하고 심각하게 실패한다. 더 결정적으로, ConDefects가 제공하는 사람 fault location은 100\%가 단일 라인(중앙값 1줄)으로 사람 버그는 국소적 슬립이지만, AI의 WA 버그는 단일 라인으로 환원되지 않고 프로그램 전체가 틀린 접근(S2/S4 90\%)이라 ``버그 라인''을 짚을 수 없다. 이는 AI 버그가 사람 버그와 질적으로 다르며 다르게 수리되어야 함을 보인다.

\subsection{RQ3: Are LLM Bugs Systematic Within a Problem?}
한 문제에 대한 10개 생성의 버그가 같은 실패 유형으로 수렴하는지 본다. 표~\ref{tab:systematicity}에서, 버그가 2개 이상인 387개 문제의 평균 modal-share는 0.845로 버그의 84.5\%가 같은 유형이었고, 38.8\%의 문제는 모든 버그가 단일 유형이었다. 예컨대 반올림 문제에서는 대부분의 시도가 동일하게 \texttt{round()}를 썼고, connectivity 문제에서는 동일하게 칸 존재 여부로 단순화하였다. 이 측정은 거친 유형 기준의 하한이다. 더 세분된 Spec-Misinterpretation 하위 유형(S1--S5) 기준으로도 수렴은 강하여, WA 버그가 2개 이상인 362개 문제에서 평균 하위 유형 modal-share가 0.791, 완전 일치 비율이 33.4\%였다. 즉 5지선다 수준의 세분성에서도 한 문제의 오답 중 약 80\%가 같은 오해를 공유하며, 이는 모델이 문제를 ``우연히 다양하게''가 아니라 ``일관되게 같은 방식으로'' 오해함을 뒷받침한다.

\subsection{RQ5: How Well Do Existing Tools Handle LLM Bugs?}
다음으로 기존 program repair 도구가 LLM 버그를 다룰 수 있는지 본다. 네 가지 수리 패러다임(탐색·의미·학습·LLM 기반) 중 교육용 cluster 기반 수리를 대표하여 Refactory~\cite{hu2019refactory}를 통합하였다. Refactory는 입문 프로그래밍에서 다수의 정답 풀이를 control flow로 클러스터링한 뒤 버기 프로그램을 가장 가까운 클러스터에 맞춰 수리한다. 우리는 AIBugBench 과제를 Refactory의 입력 형식으로 변환하여 난이도 층화 표본에 적용하였다. Refactory는 자체 예제에서는 정상 동작했으나, 경쟁 난이도 문제의 다양하고 간결한 풀이에 대해서는 refactoring 단계가 적용되지 못하여 본 분포로 전이되지 않았다. 이는 기존 교육용 수리 도구가 LLM 버그에 직접 쓰이기 어렵다는 첫 신호다.

LLM 기반 수리 패러다임을 reference baseline으로 정량화하기 위해, 난이도 층화 표본 280개 WA 버그에 문제·버기 코드·첫 실패 테스트를 주고 수정본을 받아 전체 테스트로 재채점하였다(표~\ref{tab:apr}). 생성 모델과 같은 계열(\texttt{gpt-3.5-turbo})의 자기 수리는 1.1\%(3/280)만 성공했는데, 이는 RQ2의 systematicity가 예측하는 대로 같은 모델이 같은 오해를 재현하기 때문이다. 다른 모델(\texttt{gpt-4o-mini})은 10.7\%로 약 10배 높았으나 성공의 대부분이 쉬운 문제(26/70)에 집중되었고 hard 이상은 0\%였으며, 지배적 하위 유형 S2(170개 중 19개)와 S4(84개 중 5개)는 실패 테스트를 직접 주어도 거의 고쳐지지 않았다. 즉 테스트 통과만을 oracle로 삼는 기존 수리는 본 결함 분포에 부적합하다.

L2 풀이 다양성: cluster 기반 교육용 수리는 소수의 canonical 풀이를 가정하나 경쟁 문제는 다양한 접근을 허용하여 클러스터링이 무너진다. L3 단일 모델 재현: 생성과 동일한 모델로 수리하면 같은 오해를 재현하며, 본 실험에서 자기 수리율이 1.1\%로 붕괴함을 확인하였다. L4 테스트 단독 oracle의 과적합: 명세 없이 테스트 통과만을 목표로 하면 Test-Gaming이 발생할 수 있어 spec-grounded 접근이 필요하다. L5 난이도: 어려운 문제에서 생성 통과율이 0\%이므로 in-distribution 정답 LLM 출력이 없어 학습 기반 수리의 신호가 부족하다.

\rabox{\textbf{RA5.} 교육용 cluster 기반 수리는 본 분포로 전이되지 못하고, LLM 기반 수리도 실패 테스트를 주어도 교차 모델 10.7\%·자기 수리 1.1\%에 그치며 지배적 의미 범주(S2, S4)는 거의 고쳐지지 않는다. 다섯 가지 근본 한계가 이를 설명하며, AIBugBench는 semantics-aware하고 spec-grounded한 수리의 필요성을 드러낸다.}

\subsection{Qualitative Synthesis}
세 예시(그림~\ref{fig:example},~\ref{fig:example2},~\ref{fig:example3})는 본 벤치마크 결함이 공유하는 성질을 압축한다: 모두 문법적으로 정상이고 실행되며 그럴듯해 보이지만, 명세의 의미를 서로 다른 깊이에서 어긴다. 이 성질이 다섯 연구 질문을 관통한다. 결함은 본질적으로 의미적이고(RQ1), 사람의 국소적 슬립과 질적으로 다르며(RQ2), 한 문제 안에서 같은 오해로 수렴하고(RQ3), 난이도에 따라 단순 슬립에서 알고리즘 미인식으로 양상이 이동하며(RQ4), 테스트만을 oracle로 삼는 기존 수리가 이를 다루지 못한다(RQ5).

세 예시는 왜 기존 수리가 실패하는지도 함께 설명한다. abc230\_a(S2)에서 누락된 ``42번 이후 +1'' 규칙은 코드 어디에도 ``틀린 줄''로 나타나지 않으므로, fault localization이 의심 라인을 지목할 수 없고 국소 변형 기반 수리는 추가해야 할 분기 자체를 합성하지 못한다. abc276\_c(S4)에서는 완전 탐색을 ``previous-permutation 알고리즘''으로 대체해야 하는데, 이는 한두 토큰의 수정이 아니라 알고리즘 전면 교체이므로 대부분의 패치 공간을 벗어난다. 반면 abc226\_a(S5)는 \texttt{round} 호출을 보정하는 국소 수정으로 고칠 수 있어, 표~\ref{tab:aprsub}에서 S1{\textperiodcentered}S5의 상대적 수리 용이성과 정확히 부합한다. 즉 수리 가능성은 결함의 의미적 깊이에 반비례하며, 가장 지배적인 유형(S2, S4)이 가장 깊고 가장 고치기 어렵다. 이 일관된 그림이 곧 본 벤치마크가 측정하고자 하는 것―LLM 코드를 위한 semantics-aware하고 spec-grounded한 테스트·디버깅·수리 기법―의 필요성을 정량적으로 입증한다.

\section{Discussion}
\subsection{Implications}
연구자에게: (1) Spec-Misinterpretation이 지배적이고 systematic하므로, 명세에서 입력 범위·출력 형식·edge condition 등 의미적 제약을 추출하여 테스트 통과와 별개의 두 번째 oracle로 삼는 spec-grounded repair가 유망하다. 본 연구의 수리 baseline은 실패 테스트를 직접 주어도 지배적 의미 범주가 고쳐지지 않음을 보여, 테스트 외 신호의 필요성을 실증한다. (2) 한 문제의 오답이 같은 오해로 수렴하므로(그리고 같은 모델의 자기 수리가 1.1\%로 붕괴하므로), 그 오해를 한 번 교정하면 다수 오답을 일괄 수리할 여지가 있고, 수리 모델은 생성 모델과 분리되어야 한다. (3) fault localization 연구는 본 벤치마크의 사람 fault location과 AI 버그를 함께 사용해 사람과 AI 버그에서의 정확도를 비교할 수 있으며, AI 버그가 단일 라인으로 환원되지 않는다는 점은 라인 단위 평가 지표 자체의 재검토를 요청한다. 교육자에게: 학생이 제출한 LLM 코드의 결함은 대부분 명세 오해(특히 어려운 과제일수록 알고리즘 미인식)이므로, 피드백은 문법 수정이 아니라 ``무엇을 오해했는가''를 짚어야 한다. 실무자에게: LLM 생성 코드를 평가할 때 공개 sample이 아닌 전체 테스트를 사용하고, 생성 프롬프트가 결함 분포를 좌우함(언어 미지정 시 46\% 이탈)을 인지해야 한다.

\subsection{Why Test-Only Oracles Fall Short}
본 결과들은 하나의 일관된 진단을 가리킨다: LLM 코드의 지배적 결함은 테스트만으로는 다루기 어렵다. 첫째, 결함의 76\%가 의미적(Spec-Misinterpretation)이고 그 90\%가 핵심 조건 단순화(S2)나 패러다임 미인식(S4)이므로, 결함은 ``어느 한 줄''이 아니라 ``무엇이 맞는 동작인가''에 대한 오해에 자리한다. 둘째, 한 문제의 오답이 같은 오해로 수렴하므로(coarse modal-share 0.845, 하위 유형 0.791), 같은 모델로 재시도하면 같은 틀린 동작을 다시 만들어 낸다―이는 자기 수리율이 1.1\%로 붕괴하는 데서 직접 확인된다. 셋째, 실패 테스트를 명시적으로 제시해도 교차 모델 수리가 S2와 S4를 거의 고치지 못한다(각각 11\%, 6\%). 이 세 관찰을 종합하면, 통과해야 할 테스트는 ``현재 출력이 틀렸다''만 알려줄 뿐 ``올바른 동작이 무엇인가''는 알려주지 못한다는 결론에 이른다. 사람 버그라면 한 줄을 고치는 국소 수정으로 충분하지만, AI 버그는 프로그램이 구현하는 의도 자체가 틀려 있어 테스트 신호만으로는 그 의도를 복원할 수 없다. 명세에서 추출한 의미적 제약(입력 범위·출력 형식·edge condition)은 바로 이 결핍된 신호를 공급하는 두 번째 oracle이 될 수 있으며, AIBugBench는 이 가설을 검증하는 데 필요한 명세·전체 테스트·사람 정답을 한 자리에 모아 제공한다.

\subsection{Relation to Existing Benchmarks}
AIBugBench는 규모와 성격 모두에서 기존 벤치마크를 보완한다. Defects4J(수백 개 Java 결함)와 QuixBugs(소수의 단일 라인 버그)가 사람 버그에 집중하고 HumanEval·MBPP·SWE-bench가 코드의 통과/실패에 집중하는 데 비해, AIBugBench는 3{,}691개의 AI 버그를 같은 문제의 1{,}203개 사람 버그와 짝지어 결함 유형까지 라벨링한다. ConDefects는 leakage를 통제하였으나 자연어 명세와 AI 생성 버그가 없는데, 본 벤치마크는 거기에 정규화 명세·AI 생성물·분류 라벨을 더하여 fault localization과 repair뿐 아니라 AI 대 사람 비교와 spec-grounded repair까지 함께 지원한다. 표~\ref{tab:compare}의 일곱 차원을 모두 만족하는 것은 AIBugBench가 유일하며, 특히 ``leakage-free''와 ``human-authentic''을 동시에 보장하는 점이 다른 모든 데이터셋과 구별된다.

\subsection{Threats to Validity}
구성 타당성으로, 분류에 주관이 개입할 수 있어 명확한 기준과 생성 모델과 다른 주석자 모델(\texttt{gpt-4o-mini})을 이용한 전수 라벨링, 독립적인 두 번째 패스와의 Cohen's $\kappa=0.759$로 완화하며, 사람 검증을 통한 확증은 향후 과제로 남는다. 내적 타당성으로, 현재 AI 생성물은 단일 모델 \texttt{gpt-3.5-turbo}에 한정되며 다양한 모델로의 확장은 제공된 파이프라인으로 가능하다. 단일 모델이라는 한계에도 불구하고, 벤치마크의 문제·명세·테스트·사람 버그는 모델과 무관하므로 새로운 생성 모델은 제공된 파이프라인으로 즉시 추가할 수 있으며, leakage-free 제약을 지키려면 cutoff가 윈도우 이전인 모델을 쓰면 된다. 외적 타당성으로, 사람 진정성은 ``컨테스트 시기가 ChatGPT 이전''이라는 proxy에 의존하며 개별 제출 시각이 없어 옛 문제에 대한 후대 AI 재제출 가능성은 소수로 남는다. 복수 정답을 허용하는 special judge 문제는 정확 일치 비교로 오판될 수 있어 \texttt{meta.json}에 표시하여 사용자가 제외하거나 별도 비교기를 적용할 수 있게 한다. 결론 타당성으로, AI 대 사람 분포 차이는 큰 표본(3{,}691 대 1{,}203)에서 $p<10^{-29}$로 견고하며, 수리 baseline의 결론은 난이도 층화 280개 표본에 기반하므로 절대 수치보다 범주 간·모델 간 상대 경향으로 해석한다.

\subsection{Limitations and Future Extensions}
현재의 한계와 확장 계획은 다음과 같다. 첫째, AI 생성은 단일 모델에 한정된다. 벤치마크의 문제·명세·테스트·사람 버그는 모델과 무관하므로, 제공된 파이프라인으로 cutoff가 윈도우 이전인 다른 모델을 추가하여 모델 간 결함 분포를 비교할 수 있다. 또한 이미 학습한(leaked) 최신 프런티어 모델을 같은 문제에 적용하면 ``암기 regime''에서의 결함 양상을 leakage-free 조건과 별도로 관찰하는 보완 실험이 가능하다. 둘째, 수리 baseline은 단발성 LLM 수리와 교육용 클러스터 기반 수리에 한정되며, 탐색·의미 기반 전통 APR과 반복적·agent 기반 LLM 수리로 확장하면 RQ5의 그림이 더 완전해진다. 셋째, 하위 유형 라벨링은 LLM 주석자에 의존하므로($\kappa=0.759$) 사람 검증으로 보강할 여지가 있다. 넷째, 본 벤치마크가 동기를 부여하는 spec-grounded repair―명세에서 입력 범위·출력 형식·edge condition 등 제약을 추출하여 테스트와 별개의 두 번째 oracle로 삼는 수리―의 프로토타입 구현과 평가가 자연스러운 다음 단계이며, 본 데이터셋의 명세와 짝지어진 사람 정답이 그 학습·평가 신호를 제공한다.

\section{Conclusion}
AIBugBench는 모델 cutoff와 ChatGPT 출시 사이의 겹치는 구간을 이용하여, data leakage가 없고 같은 문제에서 진정한 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크를 제공한다. 정규화 명세, 전체 hidden test, fault location, AI와 사람 버그를 함께 담아 bug classification, fault localization, automatic program repair, AI 대 사람 비교, spec-grounded repair의 다섯 태스크를 지원한다. 특징 분석에서 결함의 76\%가 Spec-Misinterpretation이고(그 90\%가 핵심 조건 단순화와 패러다임 미인식, $\kappa=0.759$) 한 문제의 버그가 강하게 수렴함을 보였다. 또한 짝지어진 사람 버그와 분포가 유의하게 다르고($\chi^2=139.8$, $p<10^{-29}$) 사람 버그는 단일 라인 슬립인 반면 AI 버그는 전체-접근 오류였으며, 기존 교육용 수리뿐 아니라 LLM 기반 수리도 실패 테스트를 주어도 교차 모델 10.7\%·자기 수리 1.1\%에 그침을 확인하였다. 방법론적으로, 본 논문의 핵심은 ``cutoff와 release의 분리''라는 단순한 관찰을 leakage-free와 human-authentic을 동시에 만족하는 구성 원리로 전환한 데 있다. 이 원리는 특정 모델이나 시기에 묶이지 않으므로, 새로운 생성 모델이 등장할 때마다 그 cutoff 이전의 컨테스트로 같은 절차를 반복하여 벤치마크를 갱신할 수 있다. 즉 AIBugBench는 한 번 만들어 고정되는 데이터셋이 아니라, LLM이 진화하는 동안 leakage 없이 따라가며 측정할 수 있는 ``살아 있는'' 평가 틀이다.

향후 작업은 세 방향이다. 첫째, 본 파이프라인으로 여러 세대의 모델을 추가하여 결함 분포가 모델 능력에 따라 어떻게 이동하는지(예: 패러다임 미인식이 줄어드는지) 추적한다. 둘째, 탐색·의미 기반 전통 APR과 반복적 LLM 수리를 baseline에 더해 RQ5의 수리 지형을 완성한다. 셋째, 본 벤치마크가 동기를 부여한 spec-grounded repair―명세에서 추출한 제약을 두 번째 oracle로 삼는 수리―를 구현·평가하여, 테스트만으로는 닫히지 않던 의미적 결함을 닫을 수 있는지 검증한다. 이러한 확장을 통해 AIBugBench는 LLM 코드의 테스트·디버깅·수리 기법을 개발하고 공정하게 평가할 지속적 토대를 제공한다.
