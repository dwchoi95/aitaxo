\title{AIBugBench: data leakage 없이 사람과 짝지은 LLM 생성 코드 결함 벤치마크}

\maketitle

\begin{abstract}
LLM(Large Language Model)이 작성한 코드가 교육과 산업 전반으로 빠르게 들어오면서, 코드 결함을 만드는 주체가 사람에서 LLM으로 이동하고 있다. 이에 따라 LLM이 만드는 버그가 사람 버그와 같은 분포인지, 사람 버그를 전제로 설계된 기존 fault localization과 program repair 도구가 LLM 버그에도 효과적인지가 중요한 질문이 되었다. 그러나 기존 코드 벤치마크는 대체로 손으로 만든 합성 과제나 임의로 선택된 데이터에 의존하며, 코드의 pass/fail 정확도만 측정할 뿐 어떤 종류의 결함이 발생하고 그것이 자동으로 탐지·수리될 수 있는지는 특징화하지 않는다. 더 근본적으로, LLM 버그가 사람 버그와 다른지를 타당하게 비교하려면 생성 모델 입장에서 data leakage가 없으면서 동시에 비교 대상 사람 코드가 진정해야 하지만, 전자는 최신 문제를 후자는 과거 코드를 요구하여 충돌한다. 그 결과 이러한 비교를 가능하게 하는 결함 벤치마크가 부재하다. 본 논문은 data leakage가 없고 같은 문제에서 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench를 제시한다. 핵심 설계는 생성 모델의 knowledge cutoff와 대화형 LLM 코딩 보조 도구의 대중적 등장 사이의 겹치는 구간을 선택하여, 대상 문제는 모델이 학습하지 못했고 그 시기의 사람 제출은 진정함을 동시에 보장하는 것이다(구체적 모델과 시기는 방법론에서 기술한다). AIBugBench는 428개 문제 각각에 대해 정규화된 자연어 명세, 전체 hidden test, LLM이 생성한 10개의 솔루션과 채점 라벨, 그리고 같은 문제에서 짝지어진 사람 정답과 사람 버그(fault location 포함)를 제공한다. open coding으로 결함을 4개 defect type과 11개 sub type으로 분류하고 두 평가자의 독립 분류로 신뢰도를 보고한다. 특징 분석에서 결함의 76\%가 Spec-Misinterpretation이며 Hallucinated-API 같은 고전적 범주는 거의 없었고, 한 문제에 대한 버그가 같은 유형으로 강하게 수렴함을 확인하였다. 또한 기존 교육용 program repair 도구가 이 결함 분포로 전이되지 못함을 보인다. 데이터셋, 코드, 재현 파이프라인을 공개한다.
\end{abstract}

\begin{IEEEkeywords}
benchmark, LLM-generated code, bug dataset, data leakage, automatic program repair, fault localization
\end{IEEEkeywords}

\section{Introduction}
LLM(Large Language Model)이 작성한 코드가 교육과 산업 전반으로 빠르게 확산되고 있다. 프로그래밍 교육에서는 학생이 과제 코드를 LLM으로 생성해 제출하는 일이 흔해졌고, 산업에서는 코딩 보조 도구가 일상화되었다~\cite{chen2021codex,tambon2024bugs}. 그 결과 코드 결함을 만드는 주체가 사람에서 LLM으로 옮겨가고 있으며, 이는 두 가지 실천적 질문을 제기한다. 첫째, LLM이 만드는 버그는 사람 버그와 같은 분포인가? 둘째, 사람 버그를 전제로 설계된 기존 fault localization과 APR(Automatic Program Repair) 도구가 LLM 버그에도 효과적인가? 이 질문들에 답하려면 LLM 코드 결함을 특징화하고 같은 문제에서 사람 버그와 비교할 수 있는 데이터가 필요하다.

이 질문의 답은 단지 학술적 호기심을 넘어선다. 교육 현장에서는 교수자가 학생이 제출한 LLM 코드의 결함을 진단하고 피드백을 주어야 하며, 결함의 유형과 그 수리 가능성을 알아야 적절한 도구와 교습 전략을 선택할 수 있다. 산업에서는 코딩 보조 도구가 만든 코드가 코드베이스에 직접 들어오므로, 그 결함 분포를 이해하는 것이 코드 리뷰와 테스트 자동화의 우선순위를 정하는 데 도움이 된다. 그러나 잘못된(예: contamination된) 데이터로 도출한 결론은 이러한 의사결정을 오도할 수 있다.

그러나 기존 코드 벤치마크는 이 목적에 잘 맞지 않는다. HumanEval~\cite{chen2021codex}과 MBPP~\cite{austin2021mbpp}는 손으로 만든 합성 과제로 사람 버그가 없고, 코드가 맞는지/틀리는지의 pass/fail만 측정할 뿐 어떤 종류의 결함이 왜 발생하는지는 특징화하지 않는다. EvalPlus~\cite{liu2023evalplus}는 테스트를 보강하면 LLM 코드의 통과율이 크게 하락함을 보였으나 여전히 실패를 분류하지 않으며, 공개 벤치마크의 contamination 문제도 보고된다~\cite{golchin2024contamination}. Defects4J~\cite{just2014defects4j}, CodeNet~\cite{puri2021codenet}, ConDefects~\cite{wu2023condefects} 같은 결함 데이터셋은 사람 버그를 담지만 AI 생성 버그를 담지 않는다. 따라서 LLM 코드의 결함을 분류하고 기존 도구로 평가할 표준 데이터가 없다.

더 근본적으로, ``LLM 버그가 사람 버그와 다른가''를 타당하게 비교하려면 두 가지 교란을 제거해야 한다. 첫째는 data leakage다. 대상 문제가 모델 학습 데이터에 이미 있으면 생성은 추론이 아니라 암기 재현이 되어, 관찰된 결함이 모델의 실제 추론 특성을 대표하지 못한다. 둘째는 사람 대조군 오염이다. LLM이 대중화된 이후의 사람 제출은 그 자체가 AI 보조의 산물일 수 있어 ``사람'' 대조군의 진정성이 훼손된다. 누수를 피하려면 최신(모델 cutoff 이후) 문제를, 진정한 사람 코드를 얻으려면 과거(LLM 대중화 이전) 코드를 써야 하므로, 같은 문제 집합에서 두 요구가 충돌한다. \textbf{그 결과, data leakage가 없으면서 진정한 사람 버그와 짝지어진 LLM 결함 벤치마크는 부재하다.}

이 공백을 메우는 벤치마크는 세 조건을 동시에 만족해야 한다. 첫째 AI 생성 버그를 포함하고, 둘째 생성 모델 입장에서 leakage가 없으며, 셋째 같은 문제에서 진정한 사람 버그와 짝지어져야 한다. \textbf{현존하는 벤치마크 중 이 세 조건을 모두 만족하는 것은 없다.}

본 논문은 leakage가 없고 사람과 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench를 제시한다. 핵심 관찰은 두 제약이 서로 다른 산출물에 서로 다른 시점으로 걸린다는 것이다. leakage는 생성 대상 문제에(모델 cutoff 기준), 사람 진정성은 사람 제출에(LLM 대중화 시점 기준) 걸린다. 두 시점이 다르므로 그 사이에 두 조건이 동시에 성립하는 겹치는 구간이 존재하며, 본 논문은 이 구간을 선택해 leakage-free와 human-authentic을 동시에 달성한다. 구체적 생성 모델과 시기 경계는 방법론에서 기술한다.

AIBugBench를 구축하기 위해, ConDefects~\cite{wu2023condefects}에서 이 구간의 Python 과제 428개를 선별하고, 각 과제의 자연어 명세를 AtCoder에서 수집해 정규화하였으며, 전체 hidden test를 확보하였다. 각 문제에 대해 LLM으로 10개의 솔루션을 생성하여 전체 테스트로 채점하고, 실패한 코드를 사람 정답과 대조하는 open coding으로 분류 체계를 도출하였다. Defects4J~\cite{just2014defects4j}와 ConDefects의 정신을 따라, 버그와 정답 버전, 테스트, 명세를 손쉽게 가져와 실행할 수 있는 재현 파이프라인을 함께 제공한다.

본 연구에서 답하는 연구 질문은 다음과 같다.

\noindent\textbf{RQ1: LLM 생성 프로그램에는 어떤 종류의 결함이 흔히 나타나는가?} 3{,}691개 버그를 분석한 결과, 결함의 76\%가 Spec-Misinterpretation으로 문법적으로 정상이고 실행되지만 다른 문제를 푸는 코드였다. 비효율로 인한 TLE(Time Limit Exceeded)가 14\%, RE(Runtime Error)가 9\%, 출력 잘림 등 CE(Compile Error)가 1\%였으며, Hallucinated-API는 사실상 없었다.

\noindent\textbf{RQ2: 한 문제에 대한 LLM의 버그는 무작위인가 systematic한가?} 버그가 2개 이상인 387개 문제에서 평균 modal-share가 0.845, 완전 일치 문제 비율이 38.8\%로, LLM은 한 문제를 일관되게 같은 방식으로 오해하였다.

\noindent\textbf{RQ3: 기존 수리 도구는 이러한 LLM 버그를 얼마나 다룰 수 있는가?} 생성 모델의 전체 통과율은 13.8\%이고 어려운 문제에서는 0\%였으며, 교육용 cluster 기반 수리 도구는 경쟁 난이도와 다양한 풀이를 가진 본 분포로 전이되지 못하였다. 우리는 기존 도구가 실패하는 다섯 가지 근본 한계를 분석한다.

본 논문의 기여는 다음과 같다.
\begin{itemize}
\item leakage가 없고 같은 문제에서 진정한 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크 AIBugBench.
\item 428개 문제, 4{,}280개 AI 생성물(문제당 10개)과 채점 라벨, 1{,}204개의 짝지어진 사람 버그(fault location 포함), 정규화 명세, 전체 hidden test로 이루어진 데이터셋과, 4개 defect type 및 11개 sub type의 분류 체계.
\item 벤치마크가 지원하는 다섯 평가 태스크와 측정 지표, 그리고 표준 분할과 평가 스크립트로 이루어진 평가 프로토콜.
\item 생성 모델의 난이도별 통과율과 결함 분포, systematicity를 reference 특성으로 보고하고, 교육용 수리 도구의 전이 실패와 그 근본 한계를 분석.
\item 데이터, 코드, 재현 파이프라인을 담은 replication package를 \url{https://github.com/dwchoi95/aitaxo} 에 공개.
\end{itemize}

\section{Background and Related Work}
\subsection{LLM Code Generation and Its Evaluation}
Codex와 그 평가 벤치마크 HumanEval~\cite{chen2021codex}, 그리고 MBPP~\cite{austin2021mbpp} 이후 LLM 기반 코드 생성은 빠르게 발전하였고, 표준 지표로 pass@$k$가 자리 잡았다. 그러나 이 지표는 코드가 맞는지/틀리는지만 보고하고 어떤 종류의 결함이 왜 발생하는지는 알려주지 않는다. 더욱이 통과/실패 판정은 테스트 스위트의 강도에 민감하다. EvalPlus~\cite{liu2023evalplus}는 HumanEval의 테스트를 대폭 보강하면 여러 모델의 통과율이 크게 하락함을 보여, 빈약한 테스트가 정확도를 과대평가함을 입증하였다. AIBugBench는 이 관찰을 결함 분석 맥락에서 재현하면서(공개 sample이 아닌 전체 hidden test 사용), 통과/실패를 넘어 실패 코드를 분류하고 같은 문제에서 사람 버그와 짝짓는다는 점에서 평가 벤치마크와 구분된다.

\subsection{Empirical Studies of Bugs in LLM-Generated Code}
LLM 코드의 결함을 경험적으로 분석한 연구가 늘고 있다. Tambon 등~\cite{tambon2024bugs}은 LLM 생성 코드의 버그 유형을 조사하여 다양한 결함 범주를 보고하였고, Jesse 등~\cite{jesse2023bugs}은 LLM이 사람과 유사한 단순 버그(simple, stupid bugs)를 생성함을 보였다. 이러한 연구는 LLM 결함의 존재와 다양성을 보였으나, 대체로 leakage 통제나 같은 문제에서의 진정한 사람 대조군 짝짓기를 명시적으로 다루지 않으며, 재사용 가능한 실행 벤치마크 형태로 공개되지도 않았다. AIBugBench는 leakage-free 구간과 짝지어진 사람 버그를 함께 확보하고, 한 문제 안에서 버그가 수렴하는 systematicity를 정량화하며, 표준 분할과 평가 스크립트를 갖춘 실행 벤치마크로 공개한다.

\subsection{Data Leakage and Contamination}
공개 벤치마크가 학습 데이터에 포함되는 contamination은 LLM 평가의 타당성을 훼손하며, 이를 탐지하고 정량화하려는 연구가 활발하다~\cite{golchin2024contamination}. 코드 생성에서 contamination은 특히 심각한데, 모델이 정답을 암기했다면 측정된 능력이나 결함 특성이 실제 추론을 대표하지 못하기 때문이다. 모델 cutoff 이후에 출제된 문제만 사용하는 것이 가장 직접적인 완화책이며, ConDefects~\cite{wu2023condefects}는 이 원리로 fault localization과 program repair 평가의 leakage를 통제하였다. 본 논문은 여기서 한 걸음 더 나아가, 생성 모델의 leakage뿐 아니라 사람 대조군의 진정성까지 동시에 보장하는 시기 윈도우를 도입한다.

\subsection{Automatic Program Repair and Fault Localization}
APR은 탐색 기반, 의미 기반, 학습 기반으로 발전해 왔으며~\cite{monperrus2018survey}, 최근에는 LLM을 직접 수리에 활용하는 접근이 강력한 성능을 보인다~\cite{xia2023llmapr}. 수리의 전 단계인 fault localization은 의심 위치를 순위화하는 오랜 연구 주제로, 다양한 기법과 평가 지표가 정립되어 있다~\cite{wong2016flsurvey}. 이들 도구는 대체로 버그가 사람에 의해 작성되었다고 가정하며, 그 결함 분포에 맞춰 설계되거나 학습된다. AIBugBench는 이들 도구를 LLM 버그에 대해 평가할 수 있는 기반을 제공하고, ConDefects가 제공하는 사람 fault location을 fault localization의 정답으로 함께 제공한다.

\subsection{Program Repair for Programming Education}
교육용 입문 프로그래밍 수리에서는 같은 과제에 대한 다수의 정답 풀이를 활용하는 접근이 주류다. Clara~\cite{gulwani2018clara}는 정답을 클러스터링하여 학생 코드를 가장 가까운 정답으로 수리하고, Refactory~\cite{hu2019refactory}는 정답 풀이 군을 control flow에 따라 재구성하여 버기 프로그램에 맞춰 패치를 합성하며, Verifix~\cite{ahmed2022verifix}는 참조 풀이에 대해 검증된 수리를 생성한다. 이들은 사람 버그와 풍부한 정답 풀이 풀을 전제로 한다는 공통점이 있다. 본 연구의 결과는 LLM 버그가 의미적으로 틀리고 풀이가 다양한 경쟁 난이도 분포를 가져, 이들 도구의 가정과 어긋날 수 있음을 시사한다.

\subsection{Defect Benchmarks}
Defects4J~\cite{just2014defects4j}는 실세계 Java 결함의 버기/패치 버전을 손쉽게 체크아웃해 controlled study를 가능하게 한 대표적 실행 벤치마크로, 이후 수많은 도메인에서 유사한 벤치마크의 모범이 되었다. CodeNet~\cite{puri2021codenet}과 Codeforces 계열은 사람 제출(정답과 오답)을 대규모로 제공하나 AI 생성 버그가 없고 LLM 이전에 구축되어 생성 모델 입장에서 leakage가 있다. ConDefects~\cite{wu2023condefects}는 모델 cutoff 이후의 AtCoder 문제만 모아 leakage를 통제하였으나 자연어 명세와 AI 생성 버그를 포함하지 않는다. \textbf{즉 LLM 생성 버그를, leakage가 없는 동시에 진정한 사람 버그와 짝지어 실행 가능한 형태로 제공하는 결함 벤치마크는 부재하며, AIBugBench가 이 공백을 메운다.}

\subsection{Selecting the Leakage-Free, Human-Authentic Window}
생성 모델로는 knowledge cutoff가 잘 문서화되어 있고 그 cutoff가 대화형 코딩 보조 도구의 대중적 등장보다 앞서는 모델이 필요하다. 우리는 \texttt{gpt-3.5-turbo}(knowledge cutoff 2021년 9월)를 고정한다. data leakage가 없으려면 대상 문제가 2021년 9월 이후 출제여야 한다. 한편 사람 대조군이 진정하려면 사람 제출이 LLM이 대중화되기 전에 작성되어야 하는데, 우리는 대중화의 분기점으로 ChatGPT의 공개 출시(2022년 11월 30일)를 택한다. ChatGPT는 학생과 개발자가 코드 생성을 위해 LLM을 일상적으로 쓰기 시작한 가장 뚜렷한 변곡점이기 때문이다. 두 시점이 다르므로 그 사이의 [2021-09, 2022-11] 구간에서 두 조건이 동시에 성립한다. 이 구간에 출제된 문제는 모델이 학습하지 못했고, 그 컨테스트의 사람 제출은 ChatGPT 이전이므로 진정하다. 다만 사람 진정성은 ``컨테스트 시기''라는 proxy에 의존하며, 옛 문제에 대한 후대 재제출 가능성은 한계로 남는다(Section~VIII).

\subsection{Problem Selection from ConDefects}
ConDefects는 각 과제를 \texttt{Code/<task>/<lang>/<id>/} 아래에 사람 버기 버전, 그에 대응하는 사람 정답, 그리고 fault location으로 제공한다. 우리는 다음 기준으로 문제를 선별한다: (i) Python 과제일 것, (ii) contest date가 시기 윈도우 안일 것, (iii) 사람 정답과 버기 짝이 모두 존재할 것, (iv) 전체 hidden test가 확보될 것. AtCoder는 2021년 후반부터 Beginner Contest의 마지막 문제 표시 라벨을 ``Ex''로 바꾸었는데, ConDefects는 이를 \texttt{\_h}로 표기한다. 이 불일치 때문에 테스트 디렉터리를 \texttt{\_h}로만 찾으면 22개 문제의 전체 테스트를 놓치므로, \texttt{\_h}를 ``Ex''로 매핑하여 복구하였다. 또한 표준 stdin/stdout 채점이 불가능한 interactive 문제 4개는 제외하였고, 복수 정답을 허용하는 special judge 문제는 메타데이터로 표시하여 정확 일치 비교의 한계를 명시한다. 최종적으로 85개 컨테스트에 걸친 428개 문제와, 그에 짝지어진 1{,}204개의 사람 버기 프로그램(및 동수의 사람 정답과 fault location)을 얻는다.

\subsection{Specifications}
ConDefects는 코드와 테스트는 제공하나 자연어 명세는 포함하지 않으므로, 각 과제의 영어 명세를 AtCoder 문제 페이지에서 수집한다. 명세는 \texttt{task-statement} 영역의 영어 블록에 있으며, 문제 설명·제약·입출력 형식·예시로 구성된다. 단순히 HTML에서 텍스트를 추출하면 수식 변수가 토큰마다 줄바꿈되어 가독성이 크게 떨어지므로, 우리는 문단과 예시 블록을 구조적으로 재구성하는 정규화를 적용한다: 문단은 한 줄로 합치고 예시 입출력 블록은 원형을 보존한다. 정규화된 명세는 \texttt{problem.txt}로 제공되며 생성과 spec-grounded repair의 입력이 된다.

\subsection{Tests and Execution Environment}
채점에는 ConDefects의 전체 hidden test를 사용한다. 솔루션은 각 테스트에 대해 stdin을 주입하고 stdout을 기대 출력과 비교하며, 출력 비교는 줄 단위 후행 공백과 마지막 개행을 무시하는 관용 비교를 따른다. 실행은 격리된 subprocess에서 이루어지고, 문제별 시간 제한과 메모리 상한을 적용하며 네트워크를 차단한다. AtCoder의 표준 라이브러리 가정을 반영하기 위해 채점 샌드박스에 \texttt{sympy}, \texttt{sortedcontainers}, \texttt{numpy}를 설치하여, 라이브러리 부재로 인한 거짓 RE를 방지한다. 공개 sample만으로 채점하면 통과율이 13.8\%가 아니라 23\%로 과대평가되므로, 전체 hidden test가 타당한 채점에 필수임을 확인하였다.

\subsection{Generating AI Solutions}
각 문제에 대해 \texttt{gpt-3.5-turbo}로 temperature 1.0, 문제당 10개의 솔루션을 생성한다. 프롬프트는 ``Output ONLY a complete Python 3 program''이라는 간결한 system 지시와 정규화된 명세(user 메시지)로 구성한다. 이 system 지시가 없으면 생성물의 46\%가 C++ 등 비-Python으로 이탈하는데, 이는 AtCoder 문제가 관습적으로 C++ 중심이기 때문이다. system 지시 한 줄로 이를 통제하여 본 설정의 모든 생성물이 100\% Python이도록 한다. 응답에서 코드는 fenced block로 추출하고, 재현성을 위해 모델 응답이 보고하는 정확한 snapshot을 레코드에 함께 저장한다. \texttt{gpt-3.5-turbo}는 저렴하고 빠르므로 문제당 10개 생성의 비용은 작다.

\subsection{Judging and Deriving the Taxonomy}
각 솔루션을 전체 hidden test로 채점하여 pass와 WA(Wrong Answer), TLE(Time Limit Exceeded), RE(Runtime Error), CE(Compile Error)로 분류한다. 결함 분류 체계는 사전 범주 없이 open coding으로 도출한다. TLE, RE, CE는 채점 status와 예외 정보로 직접 태깅한다. 의미 판단이 필요한 WA는 실행은 되나 출력이 틀린 경우로, 두 저자가 각 AI 버그를 같은 문제의 사람 정답(\texttt{correct.py})과 대조하여 무엇을 오해했는지 파악하고, 유사한 오해를 반복적으로 병합하여 하위 유형을 만든다. 이 과정에서 Spec-Misinterpretation은 다섯 하위 유형(S1--S5)으로, RE는 세 하위 유형으로 세분되어 전체 4개 defect type과 11개 sub type의 분류 체계가 도출된다(Section~VI-A). 범주가 확정된 뒤에는 전체 2{,}808개 WA 버그를 생성 모델과 다른 모델(\texttt{gpt-4o-mini})을 주석자로 사용하여 일관되게 하위 유형으로 라벨링하였고, 신뢰도는 독립적인 두 번째 주석 패스와의 Cohen's $\kappa$~\cite{cohen1960kappa}로 측정하였다. 300개 표본에서 $\kappa=0.759$(원시 일치도 0.86)로 목표 $\kappa\geq 0.7$을 만족하였다.

\subsection{Systematicity Metric}
LLM 버그가 무작위인지 systematic한지 평가하기 위해, 한 문제에 대한 여러 생성의 버그가 같은 실패 유형으로 수렴하는 정도를 정량화한다. 문제 $p$의 버그들을 범주 $c$별로 센 분포를 $\{n_c\}$, 버그 수를 $N_p=\sum_c n_c$, 서로 다른 범주 수를 $k_p$라 하자. modal-share를
\[
m_p \;=\; \frac{\max_c n_c}{N_p}
\]
로 정의한다. $m_p=1$이면 모든 버그가 같은 유형(완전 systematic)이다. 또한 분포의 정규화 Shannon entropy를
\[
H_p \;=\; \frac{-\sum_c (n_c/N_p)\,\log_2 (n_c/N_p)}{\log_2 k_p}
\]
로 정의하며($k_p>1$일 때), $H_p=0$이면 완전 systematic, $H_p=1$이면 균등 분산이다. 버그가 2개 이상인 문제에 대해 $m_p$와 $H_p$를 집계하여 systematicity를 보고한다.

\section{Supported Tasks and Evaluation Protocol}
AIBugBench는 다섯 태스크를 지원하며 각 태스크의 입력, 정답, 측정 지표를 표준화한다. Bug classification은 AI 버그(코드, 명세, 채점 결과)를 입력으로 결함 범주를 예측하는 태스크로, 정답 라벨에 대한 accuracy와 두 평가자 간 Cohen's $\kappa$로 측정한다. Fault localization은 버기 프로그램과 테스트를 입력으로 결함 라인을 순위화하는 태스크로, 사람 버그에 대해 제공된 fault location을 정답으로 사용하며 top-$k$ accuracy와 EXAM score로 측정한다~\cite{wong2016flsurvey}. Automatic program repair는 버기 프로그램, 전체 테스트, 참조 정답을 입력으로 패치를 생성하는 태스크로, 수리 후 전체 테스트를 통과하는 plausible 패치율과 정답과 동치인 correct 패치율을 구분해 측정한다~\cite{monperrus2018survey,xia2023llmapr}. AI 대 사람 버그 비교는 같은 문제에서 AI와 사람 버그의 범주 분포를 비교하는 태스크로, 분포 차이에 대한 $\chi^2$ 검정과 난이도별 교차표로 측정한다. Spec-grounded repair는 명세에서 추출한 의미적 제약을 테스트와 별개의 두 번째 oracle로 활용하는 수리 평가로, 일반 수리 대비 Spec-Misinterpretation 한정 수리율 향상으로 측정한다.

재현 가능한 평가를 위해 표준 분할과 실행 하네스를 함께 제공한다. 표준 분할은 문제 단위로 train/validation/test를 약 70/15/15로 나누어(대략 300/64/64 문제), 같은 문제의 솔루션이 분할을 넘나들지 않게 함으로써 누수 없는 평가를 보장한다. 모든 채점은 제공된 평가 스크립트와 동일한 시간 제한 및 라이브러리 환경에서 수행하여 결과를 비교 가능하게 한다.

\section{Evaluation}
이 절은 서론의 세 연구 질문에 답한다.

\subsection{RQ1: What Types of Bugs Do LLMs Produce?}
표~\ref{tab:taxonomy}는 3{,}691개 버그의 분류 분포이다. Spec-Misinterpretation이 2{,}808개(76\%)로 압도적이며, 다섯 하위 유형으로 세분된다: S1 출력/케이스 단순화(예: ``er''/``ist''를 항상 끝 두 글자로 출력), S2 핵심 조건 단순화(예: 칸 connectivity를 칸 존재 여부로), S3 잘못된 알고리즘/공식(예: 중복 순열을 일반 순열 공식으로), S4 패러다임 미인식(예: topological sort를 완전 탐색으로), S5 엣지/언어 의미(예: \texttt{round()}의 banker's rounding). 비효율로 인한 TLE가 504개(14\%), RE가 338개(9\%, 입력 형식 오독·인덱스 경계·수치/환경으로 세분), 출력 잘림 등 CE가 41개(1\%)였다. 특히 Hallucinated-API는 사실상 없었고, 관찰된 \texttt{ModuleNotFoundError}는 환각이 아니라 미설치 라이브러리 때문이었다. 2{,}808개 WA를 전수 라벨링하면($\kappa=0.759$) S2 핵심 조건 단순화가 57.3\%(1{,}609개), S4 패러다임 미인식이 32.7\%(918개)로 둘이 90\%를 차지하고, S3(6.5\%), S1(1.9\%), S5(1.6\%)는 드물었다.

\rabox{\textbf{RA1.} LLM 생성 결함은 76\%가 Spec-Misinterpretation으로, 문법적으로 정상이고 실행되지만 다른 문제를 푸는 코드다. 비효율(14\%), 런타임 오류(9\%), 출력 잘림(1\%)이 뒤따르며, API 환각은 사실상 없다.}

본 벤치마크가 가능하게 하는 핵심 비교로, 같은 문제에서 짝지어진 사람 버그 1{,}203개를 동일한 전체 테스트로 채점하여 AI와 분포를 대조하였다(표~\ref{tab:aihuman}). 두 분포는 유의하게 다르다($\chi^2=139.8$, $\mathit{dof}=3$, $p<10^{-29}$). 사람 버그는 91.5\%가 WA로 한 가지 실패에 집중되는 반면 AI는 TLE·RE·CE에서 사람보다 2{--}4배 높아 더 다양하고 심각하게 실패한다. 더 결정적으로, ConDefects가 제공하는 사람 fault location은 100\%가 단일 라인(중앙값 1줄)으로 사람 버그는 국소적 슬립이지만, AI의 WA 버그는 단일 라인으로 환원되지 않고 프로그램 전체가 틀린 접근(S2/S4 90\%)이라 ``버그 라인''을 짚을 수 없다. 이는 AI 버그가 사람 버그와 질적으로 다르며 다르게 수리되어야 함을 보인다.

\subsection{RQ2: Are LLM Bugs Systematic Within a Problem?}
한 문제에 대한 10개 생성의 버그가 같은 실패 유형으로 수렴하는지 본다. 표~\ref{tab:systematicity}에서, 버그가 2개 이상인 387개 문제의 평균 modal-share는 0.845로 버그의 84.5\%가 같은 유형이었고, 38.8\%의 문제는 모든 버그가 단일 유형이었다. 예컨대 반올림 문제에서는 대부분의 시도가 동일하게 \texttt{round()}를 썼고, connectivity 문제에서는 동일하게 칸 존재 여부로 단순화하였다. 이 측정은 거친 유형 기준의 하한이며, Spec-Misinterpretation 하위 유형 기준으로는 더 강한 수렴이 관찰된다.

\subsection{RQ3: How Well Do Existing Tools Handle LLM Bugs?}
표~\ref{tab:difficulty}는 생성 모델의 난이도별 통과율로, 벤치마크의 도전성과 reference baseline을 함께 보여준다. 전체 통과율은 13.8\%이며 어려운 문제에서는 0\%로, 벤치마크가 충분히 도전적임을 보인다. 또한 공개 sample만으로 채점하면 통과율이 23\%로 과대평가되므로 전체 hidden test가 필수임을 확인하였다.

다음으로 기존 program repair 도구가 LLM 버그를 다룰 수 있는지 본다. 네 가지 수리 패러다임(탐색·의미·학습·LLM 기반) 중 교육용 cluster 기반 수리를 대표하여 Refactory~\cite{hu2019refactory}를 통합하였다. Refactory는 입문 프로그래밍에서 다수의 정답 풀이를 control flow로 클러스터링한 뒤 버기 프로그램을 가장 가까운 클러스터에 맞춰 수리한다. 우리는 AIBugBench 과제를 Refactory의 입력 형식으로 변환하여 난이도 층화 표본에 적용하였다. Refactory는 자체 예제에서는 정상 동작했으나, 경쟁 난이도 문제의 다양하고 간결한 풀이에 대해서는 refactoring 단계가 적용되지 못하여 본 분포로 전이되지 않았다. 이는 기존 교육용 수리 도구가 LLM 버그에 직접 쓰이기 어렵다는 첫 신호다.

LLM 기반 수리 패러다임을 reference baseline으로 정량화하기 위해, 난이도 층화 표본 280개 WA 버그에 문제·버기 코드·첫 실패 테스트를 주고 수정본을 받아 전체 테스트로 재채점하였다(표~\ref{tab:apr}). 생성 모델과 같은 계열(\texttt{gpt-3.5-turbo})의 자기 수리는 1.1\%(3/280)만 성공했는데, 이는 RQ2의 systematicity가 예측하는 대로 같은 모델이 같은 오해를 재현하기 때문이다. 다른 모델(\texttt{gpt-4o-mini})은 10.7\%로 약 10배 높았으나 성공의 대부분이 쉬운 문제(26/70)에 집중되었고 hard 이상은 0\%였으며, 지배적 하위 유형 S2(170개 중 19개)와 S4(84개 중 5개)는 실패 테스트를 직접 주어도 거의 고쳐지지 않았다. 즉 테스트 통과만을 oracle로 삼는 기존 수리는 본 결함 분포에 부적합하다.

정성적으로, 기존 도구가 본 결함 분포에서 어려움을 겪는 다섯 가지 근본 한계를 분석한다. L1 의미 대 문법: Spec-Misinterpretation 버그는 문법적으로 정상이라 패턴/문법 기반 fault localization과 수리가 단서를 얻지 못한다. L2 풀이 다양성: cluster 기반 교육용 수리는 소수의 canonical 풀이를 가정하나 경쟁 문제는 다양한 접근을 허용하여 클러스터링이 무너진다. L3 단일 모델 재현: 생성과 동일한 모델로 수리하면 같은 오해를 재현하며, 본 실험에서 자기 수리율이 1.1\%로 붕괴함을 확인하였다. L4 테스트 단독 oracle의 과적합: 명세 없이 테스트 통과만을 목표로 하면 Test-Gaming이 발생할 수 있어 spec-grounded 접근이 필요하다. L5 난이도: 어려운 문제에서 생성 통과율이 0\%이므로 in-distribution 정답 LLM 출력이 없어 학습 기반 수리의 신호가 부족하다.

\rabox{\textbf{RA3.} 생성 모델은 전체의 13.8\%만 통과하고 어려운 문제에서는 0\%로, 벤치마크는 충분히 도전적이다. 교육용 cluster 기반 수리는 전이되지 못하고, LLM 기반 수리도 실패 테스트를 주어도 교차 모델 10.7\%·자기 수리 1.1\%에 그치며 지배적 의미 범주는 거의 고쳐지지 않는다. 다섯 가지 근본 한계가 이를 설명하며, AIBugBench는 semantics-aware하고 spec-grounded한 수리의 필요성을 드러낸다.}

\section{Discussion}
\subsection{Implications}
연구자에게: (1) Spec-Misinterpretation이 지배적이고 systematic하므로, 명세에서 입력 범위·출력 형식·edge condition 등 의미적 제약을 추출하여 테스트 통과와 별개의 두 번째 oracle로 삼는 spec-grounded repair가 유망하다. (2) 한 문제의 오답이 같은 오해로 수렴하므로, 그 오해를 한 번 교정하면 다수 오답을 일괄 수리할 수 있다. (3) fault localization 연구는 본 벤치마크의 사람 fault location과 AI 버그를 함께 사용해 사람과 AI 버그에서의 정확도를 비교할 수 있다. 실무자에게: LLM 생성 코드를 평가할 때 공개 sample이 아닌 전체 테스트를 사용하고, 생성 프롬프트가 결함 분포를 좌우함(언어 미지정 시 46\% 이탈)을 인지해야 한다.

\subsection{Threats to Validity}
구성 타당성으로, 분류에 주관이 개입할 수 있어 명확한 기준과 생성 모델과 다른 주석자 모델(\texttt{gpt-4o-mini})을 이용한 전수 라벨링, 독립적인 두 번째 패스와의 Cohen's $\kappa=0.759$로 완화하며, 사람 검증을 통한 확증은 향후 과제로 남는다. 내적 타당성으로, 현재 AI 생성물은 단일 모델 \texttt{gpt-3.5-turbo}에 한정되며 다양한 모델로의 확장은 제공된 파이프라인으로 가능하다. 외적 타당성으로, 사람 진정성은 ``컨테스트 시기가 ChatGPT 이전''이라는 proxy에 의존하며 개별 제출 시각이 없어 옛 문제에 대한 후대 AI 재제출 가능성은 소수로 남는다. 일부 special judge 문제는 정확 일치 비교로 오판될 수 있어 메타데이터로 표시한다.

\section{Conclusion}
AIBugBench는 모델 cutoff와 ChatGPT 출시 사이의 겹치는 구간을 이용하여, data leakage가 없고 같은 문제에서 진정한 사람 버그와 짝지어진 최초의 LLM 생성 코드 결함 벤치마크를 제공한다. 정규화 명세, 전체 hidden test, fault location, AI와 사람 버그를 함께 담아 bug classification, fault localization, automatic program repair, AI 대 사람 비교, spec-grounded repair의 다섯 태스크를 지원한다. 특징 분석에서 결함의 76\%가 Spec-Misinterpretation이고(그 90\%가 핵심 조건 단순화와 패러다임 미인식, $\kappa=0.759$) 한 문제의 버그가 강하게 수렴함을 보였다. 또한 짝지어진 사람 버그와 분포가 유의하게 다르고($\chi^2=139.8$, $p<10^{-29}$) 사람 버그는 단일 라인 슬립인 반면 AI 버그는 전체-접근 오류였으며, 기존 교육용 수리뿐 아니라 LLM 기반 수리도 실패 테스트를 주어도 교차 모델 10.7\%·자기 수리 1.1\%에 그침을 확인하였다. AIBugBench는 LLM 코드의 테스트·디버깅·수리 기법을 개발하고 평가할 토대를 제공한다.
