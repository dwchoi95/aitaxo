\title{AI가 만든 버그는 다르다: LLM 생성 코드 결함의 분류 체계와 Systematicity 분석}

\maketitle

\begin{abstract}
학생과 개발자가 LLM(Large Language Model)으로 생성한 코드를 제출하는 사례가 급증하면서, LLM이 만든 버그가 사람이 만든 버그와 다른지, 다르다면 기존 APR(Automatic Program Repair) 도구로 효과적으로 고쳐지는지가 중요한 질문이 되었다. 이 질문을 실증적으로 답하는 데에는 딜레마가 있다. 모델의 knowledge cutoff 이후 문제를 쓰면 data leakage는 피하지만 그 시기의 사람 제출은 그 자체가 AI 코드일 수 있어 사람 대조군이 오염되고, 이전 문제를 쓰면 사람 코드는 진정하지만 leakage가 발생한다. 본 연구는 두 제약이 서로 다른 산출물에 서로 다른 시점으로 걸린다는 점에 착안해, 생성 모델의 knowledge cutoff와 대화형 LLM 코딩 보조 도구의 대중적 등장 사이의 겹치는 구간을 활용한다(구체적 모델과 시기는 방법론에서 기술한다). 이 구간의 AtCoder 문제는 모델이 학습하지 못했고(leakage 없음), 그 시기 사람 제출은 LLM 대중화 이전이라 진정하다. ConDefects를 기반으로 428개 문제에 대해 4{,}280개의 솔루션을 생성하고 전체 hidden test로 채점한 뒤, open coding으로 데이터 기반 결함 분류 체계를 도출하였다. 분석 결과 LLM 결함의 76\%가 Spec-Misinterpretation으로, 문법적으로 정상이고 실행되지만 미묘하게 다른 문제를 푸는 코드였으며 Hallucinated-API 같은 고전적 범주는 거의 나타나지 않았다. 나아가 한 문제에 대한 여러 생성물의 버그가 같은 실패 유형으로 강하게 수렴함을 정량적으로 보였다(평균 modal-share 0.845, 문제의 38.8\%가 완전 일치). 같은 문제에서 짝지어진 사람 버그와 비교하면 분포가 유의하게 다르며($\chi^2=139.8$, $p<10^{-29}$), 사람 버그는 100\%가 단일 라인의 국소적 슬립인 반면 AI 버그는 프로그램 전체가 잘못된 접근이었다. 또한 LLM 기반 수리는 실패 테스트를 주어도 AI 버그의 10.7\%만, 생성과 동족 모델의 자기 수리는 1.1\%만 고쳐 지배적 의미 결함이 기존 수리에 저항함을 보였다. 이는 LLM 버그가 무작위가 아니라 systematic하고 사람 버그와 질적으로 다르며, 따라서 명세를 두 번째 oracle로 삼는 Spec-Grounded Repair가 유망함을 시사한다. leakage가 없고 사람과 짝지어진 데이터셋과 방법론을 공개한다.
\end{abstract}

\begin{IEEEkeywords}
LLM-generated code, bug taxonomy, automatic program repair, data leakage, spec-grounded repair
\end{IEEEkeywords}

\section{Introduction}
ChatGPT를 비롯한 LLM(Large Language Model)의 코드 생성 능력은 교육과 산업 현장 모두에서 빠르게 확산되었다~\cite{chen2021codex,austin2021mbpp}. 특히 프로그래밍 교육에서는 학생이 과제 코드를 LLM에게 생성시켜 제출하는 일이 흔해졌고, 이로 인해 LLM이 만든 코드 고유의 버그 패턴이라는 새로운 연구 대상이 등장하였다. 기존 APR(Automatic Program Repair) 도구는 대체로 사람이 작성한 버그를 전제로 설계되었기 때문에~\cite{monperrus2018survey,hu2019refactory}, LLM 버그가 사람 버그와 다른 분포를 가진다면 기존 도구의 효과가 떨어질 수 있다.

이 가설을 실증적으로 검증하려면 진정한 데이터가 필요하지만 두 가지 요구가 충돌한다. 첫째, AI 생성 코드가 진짜 생성이려면 대상 문제가 모델의 knowledge cutoff 이후에 출제되어 모델이 정답을 암기하지 않아야 한다(data leakage 회피)~\cite{wu2023condefects,golchin2024contamination}. 둘째, 비교 대상인 사람 코드가 진정하려면 LLM 대중화 이전에 작성되어야 한다. cutoff 이후 시기에는 사람 제출조차 AI 보조일 수 있어 대조군이 오염되기 때문이다. 전자는 ``이후''를, 후자는 ``이전''을 요구하므로 같은 문제 집합에서 두 조건을 동시에 만족하기 어렵다.

본 연구의 핵심 통찰은 두 제약이 서로 다른 산출물에 서로 다른 시점으로 걸린다는 것이다. leakage는 생성 대상 문제에 걸리고(모델 cutoff 기준), 사람 진정성은 사람 제출에 걸린다(LLM 대중화 시점 기준). 두 시점이 다르므로 그 사이에 두 조건이 동시에 성립하는 겹치는 구간이 존재한다. 본 연구는 이 구간을 활용하여 leakage-free와 human-authentic을 동시에 달성하며, 구체적 생성 모델과 시기 경계는 방법론에서 기술한다. 이 구간에 출제된 AtCoder 문제는 모델이 학습하지 못했고, 그 컨테스트의 사람 제출은 LLM 대중화 이전이므로 진정하다.

본 논문의 기여는 다음과 같다. 첫째, cutoff와 release 사이의 겹치는 구간을 이용해 위 딜레마를 해소하고 같은 문제에서 AI와 사람 결함을 짝지어 비교할 수 있게 하는 leakage-free이면서 human-authentic한 실험 설계를 제시한다. 둘째, 428개 문제와 4{,}280개 생성물의 실패를 open coding으로 분류하여 Spec-Misinterpretation이 76\%로 지배적이며 Hallucinated-API는 거의 없음을 보이는 데이터 기반 결함 분류 체계를 제시한다. 셋째, 한 문제의 여러 생성 버그가 같은 실패 유형으로 수렴함을 보여(평균 modal-share 0.845) LLM 버그가 무작위가 아닌 systematic한 현상임을 입증한다. 넷째, 문제 명세, 전체 hidden test, AI 버그(문제당 10개), 짝지어진 사람 버그(fault location 포함)를 공개한다.

\section{Related Work}
\subsection{LLM Code Generation and Evaluation}
Codex와 그 평가 벤치마크 HumanEval~\cite{chen2021codex}, 그리고 MBPP~\cite{austin2021mbpp} 이후 LLM 기반 코드 생성은 빠르게 발전하였다. 그러나 단순 통과/실패 지표는 빈약한 테스트로 인해 정확도를 과대평가하기 쉽다. EvalPlus~\cite{liu2023evalplus}는 테스트를 대폭 보강하면 LLM 코드의 통과율이 크게 하락함을 보였다. 본 연구도 공개 sample만으로 채점하면 통과율이 23\%로 과대평가되지만 ConDefects의 전체 hidden test로는 13.8\%로 떨어짐을 확인하여 같은 관찰을 LLM 버그 분석 맥락에서 재현한다.

\subsection{Bugs in LLM-Generated Code}
LLM 코드의 결함을 경험적으로 분석한 연구가 늘고 있다. Tambon 등~\cite{tambon2024bugs}은 LLM 생성 코드의 버그 유형을 조사하였고, Jesse 등~\cite{jesse2023bugs}은 LLM이 사람과 유사한 단순 버그(simple, stupid bugs)를 생성함을 보였다. 그러나 이들 연구는 대체로 leakage 통제나 같은 문제에서의 사람 대조군 짝짓기를 명시적으로 다루지 않는다. 본 연구는 leakage-free 구간과 짝지어진 사람 버그를 함께 확보하고, 나아가 한 문제 안에서 버그가 수렴하는 systematicity를 정량화한다는 점에서 차별된다.

\subsection{Data Leakage and Contamination}
공개 벤치마크가 학습 데이터에 포함되는 contamination은 LLM 평가의 타당성을 훼손한다~\cite{golchin2024contamination}. ConDefects~\cite{wu2023condefects}는 모델 cutoff 이후에 출제된 AtCoder 문제만을 모아 fault localization과 program repair 평가에서 이 문제를 완화한 데이터셋이다. 본 연구는 ConDefects를 토대로 하되, 사람 대조군 진정성이라는 추가 제약을 시기 윈도우로 함께 다룬다.

\subsection{Automatic Program Repair and Fault Localization}
전통적 APR은 탐색 기반, 의미 기반, 학습 기반으로 발전해 왔으며~\cite{monperrus2018survey}, 최근에는 LLM을 직접 수리에 활용한다~\cite{xia2023llmapr}. 수리의 전 단계인 fault localization 역시 오랜 연구 주제이다~\cite{wong2016flsurvey}. ConDefects가 제공하는 fault location은 본 데이터셋에서 사람 버그의 위치 정답으로 활용될 수 있다.

\subsection{Program Repair for Programming Education}
교육용 입문 프로그래밍 수리에서는 Clara~\cite{gulwani2018clara}가 정답 클러스터를 이용해 학생 코드를 수리하고, Refactory~\cite{hu2019refactory}는 정답 풀이 군을 재구성하여 수리하며, Verifix~\cite{ahmed2022verifix}는 참조 풀이에 대해 검증된 수리를 생성한다. 이들은 모두 사람 버그를 전제로 한다는 공통점이 있으며, 본 연구의 결과는 LLM 버그가 이들 도구가 가정하는 분포와 다를 수 있음을 시사한다.

\subsection{Competitive Programming Datasets}
IBM Project CodeNet~\cite{puri2021codenet}과 Codeforces 계열 데이터는 사람 제출을 풍부히 제공하나, 모두 LLM 대중화 이전에 구축되어 생성 모델 입장에서 leakage 문제에서 자유롭지 못하다. 본 연구는 의도적으로 leakage가 없는 ConDefects(AtCoder)를 택한다.

\section{Methodology}
\subsection{Cutoff--Release Window Design}
생성 모델로 \texttt{gpt-3.5-turbo}(knowledge cutoff 2021년 9월)를 고정한다. data leakage가 없으려면 대상 문제가 2021년 9월 이후 출제여야 하고, 사람 대조군이 진정하려면 ChatGPT 출시(2022년 11월 30일) 이전 컨테스트여야 한다. 따라서 [2021-09, 2022-11] 구간의 문제만 사용한다. 이 구간에서 ConDefects의 Python 과제 중 사람 정답과 버그 짝, 그리고 전체 hidden test를 갖춘 428개 문제를 얻는다. AtCoder가 표시 라벨을 ``Ex''로 바꾼 마지막 문제는 ConDefects의 \texttt{\_h} 표기와 매핑하여 복구하였고, stdin/stdout 채점이 불가능한 interactive 문제 4개는 제외하였다. 짝지어진 사람 버그(fault location 포함)는 1{,}215개이다.

\subsection{Specification, Tests, and Generation}
ConDefects는 코드와 테스트는 제공하나 자연어 명세는 포함하지 않으므로, 각 과제의 영어 명세를 AtCoder에서 수집해 구조적으로 정규화하였다. 채점에는 ConDefects의 전체 hidden test(문제당 평균 약 32개)를 사용한다. 생성은 모델 \texttt{gpt-3.5-turbo}, temperature 1.0, 문제당 10개로 수행한다. 프롬프트는 ``Output ONLY a complete Python 3 program''이라는 간결한 system 지시와 정규화된 명세(user 메시지)로 구성한다. 이 system 지시 한 줄이 없으면 생성물의 46\%가 C++ 등 비-Python으로 이탈하므로, 본 설정에서 모든 생성물은 100\% Python이다.

\subsection{Taxonomy Construction}
사전 범주를 두지 않고 실패 코드를 open coding으로 분류한다. 특히 WA(Wrong Answer)는 실행은 되나 출력이 틀린 경우로, 무엇을 오해했는지 알기 위해 각 AI 버그를 같은 문제의 사람 정답(\texttt{correct.py})과 대조하였다. 자동으로 판별 가능한 TLE(Time Limit Exceeded), RE(Runtime Error), CE(Compile Error)는 채점 status와 예외 정보로 1차 태깅하고, 의미 판단이 필요한 WA는 정독으로 하위 유형을 도출한다. 범주가 확정된 뒤에는 전체 WA 버그를 일관되게 라벨링하기 위해, 생성 모델과 다른 모델(\texttt{gpt-4o-mini})을 주석자로 사용하여 문제, 버기 코드, 사람 정답을 입력으로 각 버그를 하위 유형으로 분류하였다. 신뢰도는 독립적인 두 번째 주석 패스와의 Cohen's $\kappa$~\cite{cohen1960kappa}로 측정하였으며, 300개 표본에서 $\kappa=0.759$(원시 일치도 0.86)로 목표 $\kappa\geq 0.7$을 만족하였다.

\subsection{Systematicity Metric}
LLM 버그가 무작위인지 systematic한지 평가하기 위해 문제별로 그 버그들의 실패 유형 분포를 본다. 문제 $p$의 버그 집합에서 가장 흔한 유형이 차지하는 비율을 modal-share, 유형 분포의 정규화 Shannon entropy를 비systematicity로 정의한다. modal-share가 1이면 모든 버그가 같은 유형(완전 systematic)이다.

\section{Results}
\subsection{Judging Outcomes}
4{,}280개 생성물 중 589개(13.8\%)만이 전체 hidden test를 통과하였고 3{,}691개가 버그였다. 표~\ref{tab:difficulty}에서 보듯 통과율은 난이도에 따라 급격히 떨어져 어려운 문제에서는 사실상 0\%였다. 앞서 언급했듯 공개 sample만으로 채점하면 통과율이 23\%로 과대평가되며, sample만 통과하고 hidden test에서 떨어지는 약 200개 솔루션이 보정되었다. 이는 LLM 코드 평가에 전체 테스트가 필수임을 다시 보여준다.

\subsection{Sub-types of Spec-Misinterpretation}
사람 정답과의 대조를 통해 Spec-Misinterpretation을 다섯 하위 유형으로 세분하였다. S1 output/case oversimplification은 출력 분기를 하나의 규칙으로 뭉개는 경우로, 예컨대 ``er''/``ist'' 출력을 항상 끝 두 글자로 처리한다. S2 core-condition oversimplification은 진짜 요구를 더 쉬운 대용 조건으로 대체하는 경우로, 흑칸 connectivity를 ``각 행에 칸이 있는가''로 단순화한다. S3 wrong algorithm/formula는 틀린 수학 모델을 적용하는 경우로, 중복 순열 수를 일반 순열 공식으로 계산한다. S4 failed paradigm recognition은 어려운 문제에서 필요한 알고리즘을 인식하지 못하고 완전 탐색이나 임기응변으로 대체하는 경우로, topological sort를 \texttt{permutations}로 시도한다. S5 edge/language-semantics는 구조는 맞으나 경계에서 실패하는 경우로, \texttt{round()}의 banker's rounding 때문에 절반 입력에서 틀린다. 공통적으로 AI 버그는 장황하고 주석이 많아 그럴듯해 보이지만 틀린 형태였으며, 이는 문법과 패턴에 기반한 도구가 이를 놓치기 쉬운 이유와 부합한다.

표~\ref{tab:subtypes}는 2{,}808개 WA 버그를 전수 라벨링한 분포이다. S2 core-condition oversimplification이 57.3\%(1{,}609개), S4 failed paradigm recognition이 32.7\%(918개)로, 둘이 Spec-Misinterpretation의 90\%를 차지한다. 즉 LLM의 의미적 결함은 대부분 진짜 요구를 더 쉬운 조건으로 대체하거나 필요한 알고리즘 자체를 인식하지 못하는 데서 비롯된다. 출력 형식(S1, 1.9\%), 잘못된 공식(S3, 6.5\%), 언어 의미(S5, 1.6\%)는 상대적으로 드물었다.

\subsection{Systematicity}
표~\ref{tab:systematicity}는 systematicity 지표이다. 버그가 2개 이상인 387개 문제에서 평균 modal-share는 0.845로, 한 문제의 버그 중 평균 84.5\%가 같은 실패 유형이었다. 문제의 38.8\%(150/387)는 모든 버그가 단일 유형으로 완전히 일치하였다. 예컨대 반올림 문제에서는 대부분의 시도가 동일하게 \texttt{round()}를 사용했고, connectivity 문제에서는 동일하게 칸 존재 여부로 단순화하였다. 즉 LLM은 한 문제를 일관되게 같은 방식으로 오해한다. 이 측정은 거친 유형 기준의 하한이며, Spec-Misinterpretation의 하위 유형 기준으로는 더 강한 수렴이 관찰된다.

\subsection{AI Bugs versus Human Bugs}
같은 428개 문제에서 짝지어진 사람 버그 1{,}203개를 동일한 전체 hidden test로 채점하여 같은 범주로 분류하고 AI 버그와 비교하였다(표~\ref{tab:aihuman}). 두 분포는 유의하게 다르다($\chi^2=139.8$, $\mathit{dof}=3$, $p<10^{-29}$). 사람 버그는 91.5\%가 WA로 한 가지 실패에 집중되는 반면, AI 버그는 WA가 76.1\%로 낮아지고 TLE(13.7\% 대 6.1\%), RE(9.2\% 대 2.4\%), CE(1.1\% 대 0\%)에서 사람보다 2{--}4배 높았다. 즉 AI는 사람보다 더 다양하고 심각한 방식으로 실패한다. 더 결정적으로, 결함의 위치가 다르다. ConDefects가 제공하는 사람 fault location은 100\%가 단일 라인(중앙값 1줄)으로, 사람 버그는 문제를 옳게 이해한 뒤 한 줄에서 미끄러진 국소적 슬립이다. 반면 AI의 WA 버그는 단일 라인으로 환원되지 않고 프로그램 전체가 틀린 접근을 구현한 것으로(하위 유형 S2/S4가 90\%), ``버그 라인''을 짚을 수 없다. 이 질적 차이는 본 연구의 명제—AI 버그는 사람 버그와 다르며 따라서 다르게 수리되어야 한다—를 직접 뒷받침한다.

\subsection{Repairability with an Existing LLM Repair Baseline}
AI 버그가 기존 수리로 다뤄지는지 보기 위해, 난이도 층화 표본 280개(밴드당 70개)의 WA 버그에 LLM 기반 수리를 적용하였다. 각 버그에 문제, 버기 코드, 첫 실패 테스트(입력·기대·실제)를 주고 수정본을 받은 뒤 전체 테스트로 재채점하였다(표~\ref{tab:apr}). 생성 모델과 같은 계열(\texttt{gpt-3.5-turbo})의 자기 수리는 280개 중 3개(1.1\%)만 고쳤다. 이는 systematicity의 직접적 귀결로, 같은 모델이 같은 오해를 그대로 재현하기 때문이다. 다른 모델(\texttt{gpt-4o-mini})의 수리는 10.7\%로 약 10배 높았으나 여전히 낮고, 성공의 대부분이 쉬운 문제(easy 26/70)에 집중되었으며 hard 이상에서는 0\%였다. 무엇보다 지배적 하위 유형이 수리에 저항하였다: S2는 170개 중 1개(자기)/19개(교차), S4는 84개 중 0개/5개만 고쳐졌다. 실패 테스트를 직접 주었음에도 의미적 결함의 대부분이 고쳐지지 않는다는 것은, 테스트 통과만을 oracle로 삼는 기존 수리가 AI 버그에 부적합함을 보이며 명세를 두 번째 oracle로 삼는 Spec-Grounded Repair의 필요성을 뒷받침한다.

\section{Discussion}
\subsection{Implications for Spec-Grounded Repair}
Spec-Misinterpretation이 지배적이고(76\%) systematic하다(modal-share 0.845)는 두 결과는 직접적인 수리 전략을 시사한다. 버그가 문법적으로 정상이고 실행되며 명세만 어긋난다면, 자연어 명세에서 입력 범위, 출력 형식, edge condition 등 의미적 제약을 추출하여 ``테스트 통과''와 별개의 두 번째 oracle로 삼는 Spec-Grounded Repair가 유효할 수 있다. 더욱이 한 문제의 오답들이 같은 오해로 수렴하므로, 그 오해를 한 번 교정하면 다수 오답을 일괄 수리할 여지가 있다.

\subsection{Prompt Sensitivity}
언어를 명시하지 않으면 생성물의 46\%가 C++ 등으로 이탈하였다. 이는 평가 설계 시 생성 프롬프트가 결과 분포를 크게 좌우함을 보여주며, 본 연구는 ``Output ONLY a complete Python 3 program'' 지시로 이를 통제하였다. 후속 연구는 이 프롬프트 민감도 자체를 결함 분포에 미치는 요인으로 다룰 수 있다.

\subsection{Threats to Validity}
내적 타당성으로, 단일 모델 \texttt{gpt-3.5-turbo}만을 사용하였으므로 다른 모델로의 일반화는 추후 과제이다. AtCoder의 일부 special judge(복수 정답) 문제는 정확 일치 비교로 오판될 수 있으나 그 비중은 작다. 외적 타당성으로, 사람 진정성은 ``컨테스트 시기가 ChatGPT 이전''이라는 proxy에 의존한다. 개별 제출 시각은 데이터셋에 없으므로 옛 문제에 대한 후대 AI 재제출 가능성은 소수로 남는다. 구성 타당성으로, 공개 sample만으로 채점하면 통과율이 과대평가되므로(23\% 대 13.8\%) 전체 hidden test를 사용하였다.

\section{Conclusion}
본 연구는 모델 cutoff와 ChatGPT 출시 사이의 겹치는 구간을 활용하여, data leakage가 없고 사람 대조군이 진정한 환경에서 LLM 생성 코드의 결함을 분석하였다. 결함의 76\%가 Spec-Misinterpretation이었고(그 90\%가 핵심 조건 단순화 S2와 패러다임 미인식 S4) Hallucinated-API는 거의 없었으며, 한 문제에 대한 버그들이 같은 실패 유형으로 강하게 수렴하는 systematicity를 정량적으로 확인하였다. 짝지어진 사람 버그와 비교하면 분포가 유의하게 다르고($\chi^2=139.8$, $p<10^{-29}$) 사람 버그는 100\%가 단일 라인 슬립인 반면 AI 버그는 프로그램 전체가 잘못된 접근이었다. 또한 LLM 기반 수리는 실패 테스트를 주어도 AI 버그의 10.7\%만, 생성과 동족 모델의 자기 수리는 1.1\%만 고쳤으며 지배적 의미 범주는 거의 수리되지 않았다. 이는 AI 버그가 사람 버그와 다르며 테스트만을 oracle로 삼는 기존 수리로는 부족함을 보이고, 명세를 두 번째 oracle로 삼는 Spec-Grounded Repair의 필요성을 뒷받침한다. 향후 연구로 Spec-Grounded Repair 프로토타입의 구현과 평가를 계획한다.
